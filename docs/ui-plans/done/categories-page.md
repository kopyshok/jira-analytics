# Категории задач

## Контекст

Это вход задач в систему. Пользователь здесь проводит первичный разбор: смотрит входящие задачи, назначает категории, переносит задачи в активную работу, идеи или архив, включает или исключает задачи из анализа.

Экран используется как рабочее место сортировки, поэтому визуально он должен помогать быстро принимать решения и не терять несохранённые правки.

## Текущие проблемы

- Нет явной шапки с задачей экрана. Пользователь сразу попадает в фильтры, кнопки и таблицу.
- Вкладки выглядят как обычная навигация, хотя по смыслу это рабочие очереди.
- Главный рабочий признак, категория задачи, расположен далеко справа и может обрезаться на широких таблицах.
- Кнопка массового назначения категории не объясняет общий процесс: выбрать задачи, назначить категорию, сохранить.
- Пустое состояние показывает "Нет данных", хотя для очереди "К разбору" это может быть успешный результат.
- Несохранённые изменения стоит показывать как черновик, чтобы пользователь понимал, что уже изменено и что ещё не сохранено.

## Предложение

### Вариант 1: улучшить текущую таблицу

Сохранить текущую табличную модель, но сделать экран понятнее:

- добавить заголовок "Разбор задач" и короткое описание сценария;
- добавить верхнюю сводку по очередям;
- оформить вкладки как рабочие очереди с пояснением;
- сделать колонку категории заметной и не терять её за правым краем таблицы;
- показывать панель черновика только при наличии несохранённых изменений;
- оставить вторичные действия спокойными, чтобы они не конкурировали с сохранением.

Это минимальный и самый быстрый путь.

### Вариант 2: рабочее место сортировки

Сделать экран двухколоночным:

- слева список задач и групп;
- справа постоянная панель выбранной задачи;
- в правой панели показывать варианты категории, включение в анализ и сохранение;
- массовые действия оставить над списком.

Это сильнее меняет экран, но лучше соответствует задаче первичной сортировки.

## Рекомендация

Начать с варианта 1. Он даёт быстрый визуальный выигрыш без полной перестройки сценария.

После этого можно отдельно решить, нужен ли вариант 2 для больших объёмов задач.

## Что считать готовым

- На экране есть понятная шапка с назначением раздела.
- Очереди задач визуально читаются как этапы разбора.
- Пользователь сразу видит, сколько задач требует внимания.
- Категория задачи не теряется и легко считывается.
- Несохранённые изменения подсвечены и есть понятный способ сохранить или отменить.
- Пустое состояние объясняет результат и предлагает следующий шаг.

## Ограничения

- Не менять бизнес-логику назначения категорий на первом этапе.
- Не добавлять новые сущности или настройки.
- Не ломать текущую массовую работу с выбранными задачами.
