# CI Actions Minutes — Дальнейшая оптимизация

**Цель:** снизить расход GitHub Actions минут (бюджет Free 2000 мин/мес) до близкого к нулю.

**Контекст:** На 2026-05-03 за май израсходовано 256/2000 мин. Уже сделано в `.github/workflows/ci.yml` (commit от 2026-05-03):
- `paths-ignore` для `*.md`, `docs/**`, `*.png/jpg/jpeg`, `LICENSE`
- `timeout-minutes` 10/8/20 на каждый job
- `dorny/paths-filter` per-job — backend-only push не гоняет frontend, и наоборот
- E2E переведён на `pull_request` + `workflow_dispatch` (push в main больше не тратит 5-10 мин на E2E)
- Playwright cache `restore-keys` для cache hit при правках `package-lock.json`

Ожидаемая экономия после правок: **~50%** (256 → ~128 мин/мес).

Этот документ — список дальнейших оптимизаций, упорядочен по соотношению экономия/трудозатраты.

---

## Вариант A: Сделать репозиторий публичным (★★★★★)

**Экономия:** 100% минут (Actions для public repos бесплатны без лимитов).

**Трудозатраты:** 0–4 часа.

**Что нужно:**
1. Аудит истории на секреты — частично сделан 2026-05-03 (766 коммитов, 0 утечек по префиксам Atlassian/Google/GitHub/AWS/PEM, без захардкоженных паролей и `Authorization: Basic/Bearer`).
2. Прогнать `gitleaks detect` через Docker (~150 правил против ручных ~10) — для надёжности.
3. Решить — выносить ли tenant info `itgri.atlassian.net` + `Cloud ID 604dc198-...` из хардкода фронта в `VITE_JIRA_BASE_URL`. Сейчас в:
   - `frontend/src/components/analytics/AnalyticsTable.tsx:108`
   - `frontend/src/components/analytics/AnalyticsIssueDrawer.tsx:11`
   - `frontend/src/components/projects/ProjectHeader.tsx:131`
   - README.md, CLAUDE.md, AGENTS.md, app/connectors/CLAUDE.md
4. Проверить `data/` (untracked dev DB), `exports/` (xlsx с реальными данными) — оба в `.gitignore`, но физически удалить из workspace перед публикацией нелишне.
5. Если есть желание сохранить анонимность автора — переписать упоминания `akim4ik` в `docs/superpowers/specs/`.

**Минусы:** код виден всем, корпоративные данные/имена видны, обязательная лицензия.

**Когда делать:** если нет блокеров со стороны компании.

## Вариант B: E2E только вручную (★★★★)

**Экономия:** ещё ~30-50% от текущего расхода после A1+A2 правок.

**Трудозатраты:** 5 минут.

**Что нужно:**
- В `ci.yml` убрать `pull_request` из триггеров `e2e` job, оставить только `workflow_dispatch`.
- Добавить в README/CLAUDE.md инструкцию: «перед мерджем PR запустить E2E через Actions UI».

**Минусы:** легко забыть запустить, регрессии могут уйти в main.

**Когда делать:** если PR-ов мало, ручной запуск E2E дисциплинирован.

## Вариант C: Self-hosted runner на локальной машине (★★★)

**Экономия:** 100% минут на Actions, но затраты на электричество/железо.

**Трудозатраты:** 1-2 часа на первичную настройку.

**Что нужно:**
1. Установить GitHub Actions runner на ПК (`Settings → Actions → Runners → New self-hosted runner`).
2. В `ci.yml` заменить `runs-on: ubuntu-latest` → `runs-on: self-hosted`.
3. Учесть, что runner работает только когда ПК включён → CI может зависнуть если ПК выключен в момент пуша.
4. Учесть совместимость окружения — на Windows-runner'е нужны bash-совместимые скрипты, либо оставить ubuntu через Docker.

**Минусы:** ПК должен быть включён, runner — потенциальная мишень для supply-chain если репо публичный (исполнение кода из PR).

**Когда делать:** если репо приватный + есть выделенная машина 24/7.

## Вариант D: Объединить test-backend в e2e job (★★)

**Экономия:** ~10% (убирает дублирование `pip install -r requirements.txt`).

**Трудозатраты:** 30 минут на проверку что E2E падает при поломанных pytest-тестах.

**Что нужно:**
1. В `e2e` job перед `npm run e2e` запускать `python -m pytest tests/ -q`.
2. Удалить отдельный `test-backend` job.
3. Минус: теряется параллельность — wall time PR вырастет с ~10 мин до ~13-15 мин.

**Минусы:** дольше ждать результата CI, монолитный job сложнее дебажить при падении.

**Когда делать:** если параллельность не критична и хочется ещё чуть-чуть сэкономить.

---

## Вариант E: Кэш Python virtualenv целиком (★★)

**Экономия:** ~30 секунд на каждый прогон test-backend и e2e (~5-10% общего).

**Трудозатраты:** 15 минут.

**Что нужно:**
- Заменить `cache: 'pip'` (кэширует только download cache) на полноценный кэш `~/.cache/pip` + `venv/` через `actions/cache@v4` с ключом по `requirements.txt`.

**Минусы:** мелкая экономия, добавляется сложность.

**Когда делать:** в последнюю очередь, если других вариантов не осталось.

---

## Рекомендуемый порядок действий

1. **Вариант A** (публичный репо) — единственный путь к нулевому расходу. Оценить блокеры со стороны компании.
2. Если A невозможен — **Вариант B** (E2E manual) для дополнительной экономии.
3. Если бюджет всё равно близок к лимиту — **Вариант C** (self-hosted runner).
4. **Варианты D и E** — мелкие точечные правки, делать только если 1-3 не подходят.

## Метрика проверки

После каждого варианта замерить через `Settings → Billing → Metered usage → Actions minutes` за неделю и сравнить с baseline 256 мин/мес.
