# Переподтверждение категории при переезде задачи к другому родителю

**Дата:** 2026-06-16
**Раздел:** «Категории задач» (триаж / стопка на подтверждение)

## Проблема

Категория задачи часто наследуется от родителя. Бывает: задачу создают не в том родителе, руководитель подтверждает категорию, затем задачу переносят к другому родителю — у которого категория другая. Категория «застывает» неверной, и система об этом молчит.

**Пример OS-82248:** была в родителе с категорией «Тех долг», руководитель проставил «Тех долг» вручную. Потом задачу перенесли под родителя с категорией «Технические задачи». Своя категория задачи осталась «Тех долг», но контекст сменился — никто не переспросил.

## Цель

Когда у уже подтверждённой задачи сменился родитель **и** при этом сменилась категория, которую задача получает от родителя, — вернуть задачу в стопку «на подтверждение» с пометкой «сменился родитель» и историей переезда.

## Решения, принятые на брейншторме

1. **Триггер — только смена категории родителя** (вариант B). Переезд под родителя с той же категорией игнорируем. Переезд под родителя с **пустой** категорией — считается сменой (была «Тех долг» → стало пусто = значимо).
2. **Возвращаем всех переехавших** (вариант B), включая задачи со **своей** назначенной категорией (как OS-82248). Своя категория могла остаться прежней, но родитель — и контекст — сменился, руководитель должен глянуть.
3. **Одна стопка** (вариант A): возвращаем в ту же стопку «на подтверждение», где лежит первичная триажа. На задаче — значок «сменился родитель» + подсказка-история «переехала из X (была A) в Y (B)» + фильтр «только переехавшие».
4. **Исключение:** архивные и исключённые из анализа задачи никогда не дёргаем.

## Что считаем «категорией от родителя»

Категория, которую задача получила бы, **если бы у неё не было своей назначенной** — то есть обход иерархии вверх: ближайший предок с назначенной категорией → явное переопределение → корневой эпик (scope root) → fallback «незаполненные». Это контекст задачи, независимый от её собственной отметки.

Сравниваем эту «категорию от родителя» **до** переезда (зафиксированную на момент последнего подтверждения) и **после** (текущую). Отличаются → значимый переезд.

## Механика

### Хранимое на задаче

Три новых признака на задаче:

- **`parent_changed`** (да/нет, по умолчанию нет, индекс) — пометка для значка и фильтра.
- **`category_context`** — категория от родителя, зафиксированная на момент последнего подтверждения (значение «была» в подсказке, точка отсчёта).
- **`category_context_key`** — ключ родителя-источника на момент последнего подтверждения (значение «из X» в подсказке).

Текущие «куда» (новый родитель + его категория) берём вживую — они уже есть в дереве/ответе.

### Обнаружение (в пересчёте категорий, `MappingService.recalculate_issues`)

Пересчёт категорий и так пробегает все задачи после синхронизации. Добавляем на каждой задаче:

```
current = категория от родителя (resolve_inherited)
current_parent_key = ключ текущего родителя (или пусто)

если задача архивная / исключена из анализа:
    category_context = current; category_context_key = current_parent_key
    parent_changed = нет                      # тихо обновляем точку отсчёта, не дёргаем
    дальше

если category_context пустой:                 # первая встреча — инициализация
    category_context = current; category_context_key = current_parent_key
    дальше                                     # НЕ помечаем (иначе зальёт всё при первом запуске)

если задача подтверждена (category_verified):
    если current != category_context:          # контекст сменился с последнего подтверждения
        parent_changed = да
        category_verified = нет                # назад в стопку
        # category_context / category_context_key НЕ трогаем — это «откуда/была»
    # иначе подтверждена и контекст прежний — ничего
иначе (ещё не подтверждена, висит в стопке первичной триажи):
    # точку отсчёта не двигаем; пометку «переехала» не ставим (это не повторное подтверждение)
```

**Почему инициализация через пустой `category_context`:** при выкатке у 115k существующих задач точка отсчёта пуста → первый пересчёт просто записывает текущий контекст без единой пометки. Массового флага не будет. Тяжёлый backfill в миграции не нужен.

### Новый метод резолвера

`CategoryResolver.resolve_inherited_for_issue(issue)` — тот же обход вверх, что и `resolve_for_issue`, но **пропускает** шаг «своя назначенная категория». Возвращает категорию от родителя (или fallback-код, если по цепочке ничего не настроено). Переиспользует существующий цикл обхода.

### Гашение пометки (подтверждение)

Когда руководитель подтверждает категорию задачи, пометка снимается и точка отсчёта сдвигается на текущую:

```
category_verified = да
parent_changed = нет
category_context = текущая категория от родителя
category_context_key = ключ текущего родителя
```

Точки подтверждения (всё, что ставит `category_verified=True` или назначает категорию вручную):
- назначение категории на задачу (`PUT /issues/{id}/category`);
- подтверждение задачи/поддерева (verify endpoint);
- массовое «принять подсказки» (`/issues/bulk/accept-suggestions`);
- массовый каскад (`/issues/bulk/cascade-inherit`);
- батч-смена категории (`PUT /issues/batch-category`).

Выносим в один helper `reset_parent_context(db, issue, resolver)` и зовём из всех точек.

## API

- **Дерево задач** (`GET /issues/tree` и вложенные узлы): в схему узла добавить `parent_changed`, `category_context` (код «была»), `category_context_key` (ключ «откуда»). Текущий родитель уже в дереве.
- **Bulk preview** (`/issues/bulk/preview`): в фильтр добавить `only_parent_changed` (по умолчанию нет); в элемент превью — `parent_changed`. Позволяет PM массово отобрать переехавших.

## Фронт («Категории задач», `CategoriesEditorPage`)

- **Значок** «сменился родитель» на узле дерева, когда стоит пометка. Рядом — подсказка (tooltip): «Переехала из {ключ-откуда} (была {название категории «была»}) → текущий родитель {ключ родителя} ({название текущей категории от родителя})». Названия категорий — через существующий перевод кодов в подписи.
- **Фильтр-переключатель** «только переехавшие» на вкладке стопки — показывает лишь узлы с пометкой.
- Реактивность: пометка появляется/исчезает через существующий SSE `entity_changed` (issues) после пересчёта и после подтверждения.

## Граничные случаи

- **Архивные / исключённые из анализа** — точку отсчёта тихо обновляем, пометку не ставим (решение PM на брейншторме).
- **Первое появление родителя** (была без родителя, появился) — если категория от родителя меняется относительно зафиксированной, считается переездом. Приемлемо.
- **Своя назначенная категория не меняется** при переезде — задача всё равно помечается (контекст-то сменился). Это и есть случай OS-82248.
- **Многократные переезды между синхронизациями** — точка отсчёта = последнее подтверждение, поэтому ловим итоговую смену контекста, а не каждый промежуточный шаг.
- **Откат назад** (переехала и вернулась к прежнему контексту до подтверждения) — `current == category_context`, пометки нет. Корректно.

## Тесты

- Резолвер: `resolve_inherited_for_issue` игнорирует свою категорию, отдаёт категорию предка / override / scope root / fallback.
- Пересчёт: переезд со сменой категории родителя у подтверждённой задачи → `parent_changed=да`, `category_verified=нет`, контекст «была» сохранён.
- Пересчёт: переезд без смены категории родителя → без пометки.
- Пересчёт: переезд под пустую категорию → пометка ставится.
- Пересчёт: задача со своей категорией (OS-82248) → помечается.
- Пересчёт: первая встреча (пустой контекст) → инициализация без пометки.
- Пересчёт: архивная/исключённая → без пометки.
- Подтверждение (каждая точка) → пометка снята, контекст сдвинут на текущий.
- Bulk preview: фильтр `only_parent_changed` отбирает только помеченных.

## Вне рамок (YAGNI)

- Отдельная вкладка/раздел для переехавших (выбран вариант A — одна стопка).
- Email/уведомления о переездах.
- История всех переездов задачи (храним только последнюю точку отсчёта).
- Возврат архивных/закрытых/старых задач (исключены решением PM).
