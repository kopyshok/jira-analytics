# Опросник сисадмину — публикация JiraAnalysis на корпоративном сервере

Готовы 4 фазы (CI на Postgres + Docker-образ + GHCR release-pipeline + миграция данных).
Чтобы фактически развернуть на сервере — нужны ответы на вопросы ниже. Большинство влияют на конфиги, которые мы передаём вам вместе с образом.

---

## A. Сервер и сеть

### A1. Класс машины

Рекомендованный минимум: **4 vCPU / 16 GB RAM / 100 GB SSD**.

- Что выделяете (vCPU / RAM / диск)?
- Это виртуалка или физический сервер? Если ВМ — какой гипервизор (для снапшотов)?
- ОС (ожидаем Ubuntu 22.04 LTS или RHEL/Alma 9). Версия?

**Зачем:** на 16 GB параллельно поместятся prod (postgres+backend+nginx) и staging — небольшой запас. Embedding-модель (~470 MB) живёт в RAM backend-контейнера.

---

### A2. Внутренний домен (Q4 из спека)

- Какой hostname резолвится в IP сервера для пользователей?
  Пример: `jira-analytics.itgri.local`, `jira.corp.internal`
- Это «настоящее» DNS-имя в корп. DNS или нужно добавлять?
- Будет ли staging доступен по отдельному hostname (например `jira-analytics-staging.itgri.local`) или только по IP+порт?

**Зачем:** домен прописывается в `nginx.conf` (`server_name`), в `.env` (`CORS_ORIGINS`), в TLS-сертификате.

---

### A3. Firewall — что открыто (Q9)

- Какие порты сервера доступны клиентам из корп. сети?
  Ожидаем: `443` (HTTPS) обязательно, `80` (HTTP → редирект на HTTPS) желательно.
- Доступ только из офисной подсети или через VPN тоже?
- Нужно ли согласование IP-блока пользователей для whitelisting?

---

### A4. Исходящий трафик с сервера (Q5)

Сервер должен ходить наружу как минимум сюда:

| Цель | Куда | Зачем |
|---|---|---|
| GHCR | `ghcr.io`, `pkg-containers.githubusercontent.com` (HTTPS:443) | docker pull образа на каждое обновление |
| Jira Cloud | `itgri.atlassian.net` (HTTPS:443) | синхронизация задач/ворклогов |
| Gemini API | `generativelanguage.googleapis.com` (HTTPS:443) | AI-саммари проектов (опционально) |
| OpenRouter | `openrouter.ai` (HTTPS:443) | резервный LLM-провайдер (опционально) |
| RU производственный календарь | `data.gov.ru` или мирроры (HTTPS:443) | синхронизация календаря |

Вопросы:
- Что закрыто? Если корп. прокси — какой URL/port?
- HuggingFace **не нужен** в runtime (модель встроена в образ). Уточняем чтоб не блокировали лишнего.
- Если что-то полностью закрыто — какие альтернативы (внутренний registry для образов, mirror)?

## B. TLS и обратный прокси

### B1. Источник сертификата (Q2)

Варианты:

1. **Корпоративный CA** (рекомендуем для внутреннего сервиса) — вы выдаёте `fullchain.pem` + `privkey.pem`, мы кладём в `deploy/prod/proxy/certs/`.
2. **Внешний balancer терминирует TLS** — backend принимает по HTTP, nginx у нас можно отключить.
3. **Let's Encrypt** — требует publicly-resolvable домен и доступ по 80 порту с интернета. Скорее не подходит.

- Какой вариант?
- Если вариант 1: какой срок действия и кто отвечает за продление? (chmod 600 на ключ обязательно)
- Если вариант 2: какой hostname в `Host`-заголовке backend увидит? Поддерживаются ли SSE через ваш balancer (long-lived HTTP)?

---

### B2. Обратный прокси

Мы предлагаем nginx внутри docker-compose. Альтернативы:

- На сервере уже есть nginx/traefik/HAProxy — тогда наш контейнер с nginx убираем, backend выставляем на внутренний порт, ваш прокси проксирует.

- Какой вариант предпочитаете?
- Если ваш прокси: нужен ли список путей с особыми требованиями? Особое внимание SSE-endpoints:
  - `GET /api/v1/events/stream` — long-lived, `Connection: keep-alive`, no buffering, timeout ≥ 1h
  - `POST /api/v1/sync/...` (несколько) — streaming, timeout ≥ 30 min
  - `POST /api/v1/work-type-report/build/stream` — streaming

## C. Образы и GHCR

### C1. Pull-токен для GHCR

Образ публикуется в `ghcr.io/<github_org>/jira-analytics:<vX.Y.Z>`. GHCR требует авторизации даже для public-пакетов в некоторых корп. сетях.

- Можете создать на сервере service-account на GitHub (или взять от разработчика) и сохранить токен с scope `read:packages`?
- Где будете хранить токен? (`docker login ghcr.io ...` сохраняет в `~/.docker/config.json` под пользователем).
- Альтернатива: внутренний registry (Harbor/Nexus). Если есть — какой URL и как туда пушить из GitHub Actions?

---

### C2. Имя GitHub-организации/пользователя

В `deploy/prod/.env` стоит `GHCR_ORG=akim4ik` (текущий владелец репо). Если будет миграция репо в корп. GitHub-org — поменять там и в `release.yml`.

- Куда планируем переносить репо для долгосрочного владения? Останется на личном аккаунте или переедет в `github.com/<company-org>/`?

## D. Данные и резервные копии

### D1. Снапшоты ВМ (Q6)

- Частота снапшотов и срок хранения?
- Тестировался ли restore?
- Сколько времени занимает откат (нужно знать для рассчёта downtime при ЧП)?
- Можете делать **дополнительный ручной снапшот по запросу** перед каждым релизом? (Стандартная процедура из runbook §5)

---

### D2. Резервные копии Postgres

ВМ-снапшот достаточен, но дополнительно полезен `pg_dump` (быстрее восстанавливается, можно частично восстанавливать).

- Подключите ли cron на `pg_dump` (например ежедневно в 03:00, хранение 14 дней)?
- Куда складывать дампы (локально, NAS, S3-совместимое хранилище)?
- Или этого не нужно и достаточно VM-снапшотов?

---

### D3. Передача SQLite-снимка для миграции

Однократно при первом запуске нужно перенести файл `~457 MB` с моего ПК на сервер (`docs/superpowers/specs/2026-05-16-data-migration-runbook.md`).

- Как лучше передать? Варианты:
  - SCP с моего ПК на сервер (нужен временный SSH-доступ для меня или вы скачиваете через jump-host)
  - Корп. файлообменник (внутренний S3, SharePoint, etc.)
  - SFTP / share
- После миграции файл нужно удалить с сервера — учтём в runbook.

## E. Мониторинг и алерты

### E1. Uptime-мониторинг (Q7)

- Есть ли в компании внутренний uptime-tool (Zabbix, Prometheus, Grafana Synthetic, etc.)?
- Если нет — можно поставить UptimeRobot (бесплатный, внешний); но он не сможет долбить внутренний домен без VPN/туннеля.
- Endpoint для проверки: `GET https://<домен>/health` → ожидается 200 + `{"status":"healthy"}`.
- Куда слать алерты (email, Telegram, корп. чат)?

---

### E2. Логирование

Контейнеры пишут в stdout/stderr. Docker драйвер `json-file` с ротацией (50 MB × 5 файлов = ~250 MB на сервис).

- Есть ли централизованная система логов (ELK, Loki, Splunk)? Если да — настройте docker logging driver на отправку туда.
- Если нет — `docker compose logs backend` достаточно для разбора инцидентов.

---

### F2. Хранилище секретов (Q10)

В `.env` будут:
- `DB_PASSWORD` (генерируется один раз, постоянный)
- `JWT_SECRET_KEY` (генерируется один раз, ротация = разлогин всех пользователей)
- `GEMINI_API_KEY`, `OPENROUTER_API_KEY` (можно ротировать)
- `ADMIN_PASSWORD` (временный, удаляется после первого входа)

- Есть ли корп. password manager (Vaultwarden, Bitwarden, HashiCorp Vault)? Желательно сохранить туда `JWT_SECRET_KEY` и `DB_PASSWORD` — потеря = пересоздание БД и принудительный разлогин всех.
- Куда положите свою копию? `chmod 600 .env` на сервере мы укажем, плюс рекомендуем зеркало в корп. хранилище.
