# Scenario xlsx export — нарядная и детальная книга

**Дата:** 2026-04-25
**Endpoint:** `GET /api/v1/exports/scenarios/{id}.xlsx` (существующий, переписывается под новый формат)
**Сервис:** `ExportService.build_scenario_xlsx` (полный rewrite)

## Контекст

Текущий экспорт сценария — один лист «Сценарий» с шапкой из 5 строк
(ёмкость / план / остаток / включено / пропущено) и плоской таблицей
`Задача / Приоритет / Оценка / План / Статус`. Из всего, что PM видит на
`/planning` (ResourceSummary, ScenarioRulesEditor, разбивка по ролям и
обязательным работам, отсутствия, дефицит, статус «черновик/утверждён»,
внешний QA, ОПЭ-сплит), в файле не остаётся ничего.

Цель — заменить лист на книгу из 7 листов, покрывающую три аудитории
сразу (отчёт «наверх», рабочий снимок PM, артефакт для команды). Стиль —
презентационный максимум на «обложке» и в разделах для руководства,
плотные данные без украшений на рабочих листах.

## Аудитории и роль каждого листа

| Лист | Назначение | Аудитория | Стиль |
|------|------------|-----------|-------|
| 1. Обложка | TL;DR сценария, метрики, прогресс | руководство | презентационный |
| 2. Инициативы — включено | Что в квартале, разбивка по ролям | PM + команда | плотный |
| 3. Не вошло | Что не влезло, причина | руководство + PM | плотный |
| 4. Ресурс по ролям | Ёмкость / обязательные / план / дефицит | PM | плотный + heatmap |
| 5. Ресурс по сотрудникам | Норма / отсутствия / доступно по людям | команда | плотный |
| 6. Правила и обязательные работы | Снимок ScenarioRulesEditor | PM + команда | плотный |
| 7. Отсутствия | Все absences в команде за квартал | команда | плотный |

## Стилистическая база (общая для всей книги)

- Тёмная палитра в шапках листов под UI: фон `#0F2340`, текст белый.
  Акцент cyan `#00C9C8` — для бейджей, прогресс-бара, бордюров итогов.
- Светлая палитра в данных (стандартный белый фон Excel) — чтобы файл
  читался при печати и не отнимал ресурсы Excel на 100k+ ячейках.
- Шрифт: `Calibri` (дефолт Excel, не требует встраивания).
- Заморозка строк: первые 1–2 строки (шапка) на каждом листе.
- Автофильтр на всех табличных листах.
- Сетка скрыта (`sheet_view.showGridLines = False`) — границы ячеек
  рисуем только там, где нужно.
- Колонки фиксированной ширины, рассчитанной под содержимое (без
  `bestFit` — он медленный и ненадёжный в openpyxl).
- Имя файла: `scenario_{quarter}_{year}_{name_slug}_{YYYY-MM-DD}.xlsx`,
  напр. `scenario_Q2_2026_alpha-team-base_2026-04-25.xlsx`.

### Цветовая семантика (одинаковая везде)

- **Зелёный** `#52C41A` (фон `#F6FFED`) — норма, утверждённое.
- **Жёлтый** `#FAAD14` (фон `#FFFBE6`) — предупреждение, 90–110 %.
- **Красный** `#FF4D4F` (фон `#FFF1F0`) — дефицит, перегруз > 110 %.
- **Серый** `#8C8C8C` (фон `#FAFAFA`) — пропущенное, нерелевантное.
- **Cyan** `#00C9C8` — акцент, бейджи, итоги.

**Реализация:**
- Заголовок (объединённые ячейки `A1:H1`, шрифт 24pt bold, белый по
  тёмному фону).
- Бейдж статуса — отдельная ячейка с заливкой (зелёный `approved`,
  серый `draft`).
- Карточки 3×2 — каждая карточка занимает блок 3×3 ячейки: верхняя
  строка — лейбл (10pt серый), средняя — большая цифра (28pt bold),
  нижняя — подпись (9pt). Обводка cyan.
- Прогресс-бар — две объединённые ячейки одного блока, левая часть
  заливается cyan-ом пропорционально использованию, правая — серым.
  Реализуется как один merged-cell-блок шириной 28 ячеек, заливка
  через несколько cell с одной высотой.
- Таблица «по ролям» — обычная Excel-таблица без украшений, кроме:
  итоги жирные, ячейка `% исп.` через conditional formatting (зелёный
  / жёлтый / красный по семантике).

**Источник данных:**
- `PlanningScenario` (имя, статус, год, квартал, team).
- `ResourceBaseService.summary(scenario)` → `gross_by_role`, `available_by_role`.
- Сумма `planned_hours_by_role` из allocations (новый агрегат, см. ниже).
- Время генерации — `datetime.utcnow()` в Europe/Moscow.

  (`min=0` светлый, `max` максимум по столбцу — насыщенный cyan).
  Подчёркивает «тяжёлые» задачи визуально.
- Колонка «Итого» — жирный, заливка `#F0F9FF` (бледно-cyan).
- Последняя строка — `Σ ИТОГО` по числовым колонкам (жирный, белый
  текст, фон cyan), заморожена через `freeze_panes` сверху и снизу
  (последняя строка фиксируется отдельно — openpyxl не умеет, сделаем
  обычной строкой с явным выделением).
- Автофильтр на всю таблицу (`AutoFilter`).
- Условное форматирование на «Приоритет»: 1 → красный шрифт жирный,
  2 → оранжевый, 3+ → серый.

## Лист 3: Не вошло

Те же колонки, что лист 2, но для allocations с `included_flag=False`.

**Отличия:**
- Заголовок листа: «Не вошло (N)» с подсветкой числа.
- Фон строк — `#FAFAFA` (бледно-серый, отличает от листа 2).
- Колонка **«Причина»** не добавляется. Источник причины «почему не
  вошло» в данных отсутствует (greedy-алгоритм удалён, всё снимается
  вручную). Если в будущем причина появится в модели — добавим
  колонку без структурных изменений.
- Heatmap-окраски часов нет (бессмысленна на исключённых).

## Лист 4: Ресурс по ролям

Матрица: строка = роль, колонки = метрики ресурса.

**Колонки:**

| Колонка | Источник | Формат |
|---------|----------|--------|
| Роль | `Role.label` | text, bold |
| Сотрудники в роли | `role_employee_names[role]` | comma-joined, italic |
| Норма (календарь), ч | `gross_by_role[role]` | float |
| Отсутствия, ч | `gross - calendar_with_absences` | float, серый |
| ─ Из них планов., ч | абсолют по `is_planned=True` reasons | float, серый |
| ─ Из них внепл., ч | абсолют по `is_planned=False` reasons | float, серый |
| **Обязательные работы (детально)** ─ N колонок (по числу work types с `subtracts_from_pool=True`) | `WorkTypeSummaryRow.hours_by_role[role]` | float |
| Σ Обязательные, ч | сумма колонок выше | float, bold |
| Доступно для инициатив, ч | `available_by_role[role]` | float, **bold** |
| Запланировано, ч | сумма `planned_hours_by_role` из allocations | float |
| Дефицит, ч | `available - planned` (отрицательное = дефицит) | float, conditional |
| Использование, % | `planned / available` | percent, conditional |

**Спецоформление:**
- Колонки «Обязательные работы (детально)» — динамическое число; их
  заголовок = `MandatoryWorkType.label`, под шапкой — мини-подзаголовок
  «N % нормы» (бледный шрифт, 9pt) с процентом из `ScenarioRule`.
- Heatmap на колонке «Использование, %»:
  - 0–80 % — зелёная заливка `#F6FFED`.
  - 80–100 % — жёлтая `#FFFBE6`.
  - 100–110 % — оранжевая `#FFF7E6`.
  - >110 % — красная `#FFF1F0`, текст красный bold.
- Колонка «Дефицит, ч» — стрелка ▼ (через юникод) перед отрицательным
  числом, красный bold.
- Внизу строка `Σ ИТОГО` со суммированием всех числовых колонок
  (`SUM(B2:B<row-1>)` — Excel-формула, чтобы при ручной правке листа
  итоги пересчитывались).
- Внешний QA, если задан (`scenario.external_qa_hours`), не входит в
  суммы по ролям, но добавляется отдельной строкой ниже итогов с
  пометкой «*Внешний ресурс QA*» (italic, не учитывается в %).

## Лист 5: Ресурс по сотрудникам

По строке на сотрудника, сгруппировано по командам (если их несколько).

**Колонки:**

| Колонка | Источник | Формат |
|---------|----------|--------|
| Команда | `EmployeeTeam.team` (primary) | text, в группирующей строке |
| Сотрудник | `Employee.display_name` | text |
| Роль | `Role.label` | text |
| Норма (календарь), ч | сумма `ProductionCalendarDay.hours` | float |
| Отсутствия (план.), ч | absences с `is_planned=True` | float |
| Отсутствия (внепл.), ч | absences с `is_planned=False` | float |
| Доступно, ч | `total_hours` из `EmployeeBase` | float, **bold** |
| Раб. дней в квартале | `count(days where hours > 0)` из `EmployeeBase.days` | int |
| Дней отсутствия | `count(days where absence_overlap)` из `EmployeeBase.days` | int |

**Зачем числа вместо спарклайна:** openpyxl 3.x не поддерживает
Excel sparklines (`<x14:sparklineGroups>` требует ручной правки XML
workbook). Менять библиотеку на `xlsxwriter` ради одного спарклайна
дорого — он используется во всех экспортах. Числовые колонки дают
ту же информацию (нагрузка vs. отсутствия), но без визуала.

**Спецоформление:**
- Группировка по команде: первая строка каждой группы — заливка cyan,
  жирный, имя команды + итоги по группе (формула `SUMIFS` от
  Excel, чтобы при правке пересчитывались).
- Условное форматирование на «Дней отсутствия»:
  - 0–5 — без заливки.
  - 6–15 — жёлтый фон.
  - >15 — красный фон.

## Лист 6: Правила и обязательные работы

Снимок `ScenarioRulesEditor`.

**Структура:**
1. **Шапка:** «Правила сценария: {name}». Если правила сценария
   отличаются от шаблона ролевых правил (`role_capacity_rules`) — под
   шапкой курсивная пометка «Правила переопределены вручную».
2. **Матрица «Роль × Вид работы»:**
   - Строки — роли в команде (плюс служебная строка «Все роли»
     для NULL-правил).
   - Колонки — `MandatoryWorkType` с `subtracts_from_pool=True`.
   - В ячейке — `percent_of_norm` из `ScenarioRule` + расчётные часы
     (`gross_by_role × percent / 100`) в виде «10 % · 72 ч».
   - Если правила нет — пустая ячейка (не «0»), цвет `#FAFAFA`.
3. **Сумма по строке (правее матрицы):**
   - «Σ % обязательных» — общий процент нормы, занятой обязательными
     работами для этой роли.
   - Если сумма >100 % — красный bold (это конфликт правил).
4. **Внешний QA-лимит:** отдельная строка под матрицей,
   `scenario.external_qa_hours`. Если NULL — «не задан».
5. **Виды работ, не вычитающие из пула** (`subtracts_from_pool=False`)
   — отдельная мини-таблица ниже, без часов, только название и
   процент. Пометка «не уменьшает доступное время».

**Заливка ячеек по проценту** (heatmap):
- 0–10 % — без заливки.
- 10–25 % — `#F0F9FF` (бледный cyan).
- 25–50 % — `#BAE7FF` (cyan).
- >50 % — `#69C0FF` (тёмный cyan).

## Лист 7: Отсутствия

Список всех `Absence` для сотрудников команды сценария, чьи периоды
пересекаются с кварталом.

**Колонки:**

| Колонка | Источник | Формат |
|---------|----------|--------|
| Сотрудник | `Employee.display_name` | text |
| Команда | primary `EmployeeTeam.team` | text |
| Роль | `Role.label` | text |
| Причина | `AbsenceReason.label` | text, цвет фона = `AbsenceReason.color` |
| Тип | «Плановая» / «Внеплановая» | по `is_planned` |
| Начало | `Absence.start_date` | date `DD.MM.YYYY` |
| Конец | `Absence.end_date` | date `DD.MM.YYYY` |
| Дней | разница (раб. дней по календарю) | int |
| Часов потеряно | `Absence.hours_total` (если NULL — пересчёт по `ProductionCalendarDay`) | float |

- Сортировка: команда → сотрудник → дата начала.
- Цвет фона ячейки «Причина» = `AbsenceReason.color` (HEX из БД).
  Жёсткий фон, белый текст с обводкой 1px серым.
- Условное форматирование на «Часов потеряно»: подсветка >40 ч
  жёлтым, >80 ч красным.
- В шапке листа — итог: «Всего часов потеряно: X (план.: Y, внепл.:
  Z)».

## Архитектура реализации

### Новый модуль: `app/services/scenario_xlsx_export.py`

Текущий `ExportService.build_scenario_xlsx` — 80 строк inline. Новый
формат — ~700 строк, разбитый на 7 функций (по листу). Чтобы
`export_service.py` не разрастался до 1500+ строк, выносим логику
сценарного экспорта в отдельный модуль.

Контракт модуля:

```python
class ScenarioXlsxExporter:
    def __init__(self, db: Session, scenario_id: str): ...
    def build(self) -> bytes: ...

    # Внутренние:
    def _load(self) -> ScenarioExportContext: ...
    def _sheet_cover(self, ws, ctx): ...
    def _sheet_initiatives_included(self, ws, ctx): ...
    def _sheet_initiatives_excluded(self, ws, ctx): ...
    def _sheet_resources_by_role(self, ws, ctx): ...
    def _sheet_resources_by_employee(self, ws, ctx): ...
    def _sheet_rules(self, ws, ctx): ...
    def _sheet_absences(self, ws, ctx): ...
```

`ExportService.build_scenario_xlsx(scenario_id)` становится тонким
адаптером:

```python
def build_scenario_xlsx(self, scenario_id: str) -> bytes:
    return ScenarioXlsxExporter(self.db, scenario_id).build()
```

### Загрузка данных — единый контекст

```python
@dataclass
class ScenarioExportContext:
    scenario: PlanningScenario
    allocations: list[ScenarioAllocation]      # с .backlog_item.issue eagerly loaded
    resource_summary: ResourceSummary           # из ResourceBaseService.summary()
    resource_base: ResourceBase                 # из ResourceBaseService.compute()
    scenario_rules: list[ScenarioRule]          # с work_type eagerly loaded
    work_types: list[MandatoryWorkType]         # все, упорядочены
    employees: list[Employee]                   # члены команды с teams + role
    roles_by_code: dict[str, Role]              # справочник ролей (label, color)
    absences: list[Absence]                     # за квартал, с reason eagerly
    calendar_by_date: dict[date, float]         # ProductionCalendarDay в квартале
    jira_base_url: str                          # из AppSetting для гиперссылок
    generated_at: datetime
```

Один `_load()` забирает всё одним проходом — никаких N+1.

## Изменения в API

Endpoint и сигнатура не меняются — `GET /exports/scenarios/{id}.xlsx`
возвращает `StreamingResponse` с media-type
`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`.
Только имя файла в `Content-Disposition` обновляется до
`scenario_{quarter}_{year}_{slug}_{date}.xlsx`.

## Что НЕ делаем (out of scope)

1. **Сравнение двух сценариев в одной книге.** Уже есть
   `ScenarioCompareDrawer` в UI. Если потребуется отдельный xlsx —
   будет отдельный endpoint с другой структурой.
2. **Экспорт для нескольких сценариев батчем.** PM выгружает по
   одному.
3. **Спарклайны.** openpyxl не поддерживает; добавлять `xlsxwriter`
   ради этого — слишком дорого. Вместо них — числовые колонки.
4. **PDF-копия книги.** PPTX-экспорт остаётся в текущем виде, его не
   трогаем в этой задаче.
5. **Локализация.** Все подписи на русском, как и UI. Английская
   версия не нужна.
6. **Динамический выбор листов.** Книга всегда из 7 листов. Если
   данных нет (нет отсутствий, нет правил) — лист всё равно
   создаётся, но содержит пометку «нет данных».
7. **Обновление PPTX-экспорта.** Текущий `build_scenario_pptx`
   живёт своей жизнью; если позже потребуется выровнять стили —
   отдельная задача.

## Тестирование

Бэкенд (pytest, `tests/test_export_service.py` + новый
`tests/test_scenario_xlsx_export.py`):

1. **Smoke** — `build()` для пустого сценария (нет allocations, нет
   правил) возвращает валидный xlsx с 7 листами.
2. **Cover sheet** — проверить, что метрики совпадают с расчётами
   `ResourceBaseService.summary` и суммой allocations.
3. **Included / excluded** — корректно делятся allocations по
   `included_flag`; «Итого» совпадает с реальной суммой.
4. **By role** — для роли с явным `ScenarioRule` процент совпадает с
   правилом; для роли без правила процент = 0.
5. **By employee** — отсутствия суммируются по правильным дням
   календаря, не дублируются при пересечении периодов.
6. **Rules sheet** — переопределённые правила помечаются курсивом;
   external_qa_hours рендерится в отдельной строке.
7. **Absences sheet** — попадают только absences, пересекающиеся с
   кварталом; цвет ячейки «Причина» совпадает с `reason.color`.
8. **Conditional formatting** — проверять не визуал, а наличие правил
   (`ws.conditional_formatting` непустой).

Без E2E (Playwright) — экспорт скачивается, но визуальная проверка
xlsx из браузера не нужна; достаточно убедиться, что endpoint
возвращает 200 + правильный `Content-Type`.
