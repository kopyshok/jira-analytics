# Дашборд руководителя

Раздел `/executive` — обзор для руководителя сопровождения 1С на одной странице. KPI верхнего уровня, тренды, риски и ИИ-сводка из трёх блоков. Не подменяет Аналитику или Capacity, а собирает из них короткую выжимку «на что смотреть в первую очередь».

---

## 1. Зачем раздел

Руководителю не нужно каждый раз нырять в Аналитику, Capacity и Бэклог по отдельности. Дашборд руководителя отвечает на четыре вопроса:

1. **Как мы себя чувствуем в целом?** — индекс здоровья.
2. **Хватает ли людей?** — загрузка ресурсов по ролям.
3. **Что горит?** — критичные риски.
4. **Идём ли по плану?** — выполнение утверждённого сценария квартала.

Сверху четыре числа, ниже — ИИ-сводка «что улучшилось / где риск / что делать», ниже — графики и список топ-рисков. Заходишь на пять минут, понимаешь картину, решаешь куда углубиться.

---

## 2. Когда сюда заходить

- Утром понедельника — обзор за прошлую неделю.
- Перед ПМ-комитетом — быстрая выжимка по квартальному прогрессу и блокерам.
- Когда из Аналитики или Capacity всплыла подозрительная цифра — посмотреть, виден ли тренд на верхнем уровне.
- Перед регулярной встречей с заказчиком — взять оттуда «здоровье модулей» и «топ рисков».
- Когда нужно показать кому-то «как у нас дела», а времени на словесный пересказ нет — переключить квартал и показать страницу.

Раздел не для ежечасного мониторинга. Это снимок, который собирается по запросу.

## 3. Элементы интерфейса

| № | Блок | Что показывает |
|---|------|----------------|
| 1 | Шапка | Название периода + дата последней сборки ИИ-сводки + кнопка «Построить / Пересчитать» |
| 2 | Глобальный фильтр команды (в шапке приложения) | Сужает данные до выбранных команд |
| 3 | Глобальный пикер периода (в шапке приложения) | Квартал и год, за которые собирается дашборд |
| 4 | Четыре карточки KPI | Индекс здоровья, загрузка ресурса, критичные риски, выполнение сценария |
| 5 | ИИ-сводка (три плашки) | «Что улучшилось» / «Где риск» / «Что делать» |
| 6 | Тренд здоровья (8 недель) | Как менялся индекс здоровья по неделям |
| 7 | Здоровье модулей | Список функциональных модулей с маркером здоровья и загрузкой |
| 8 | Очередь по типам | Количество открытых задач по приоритетам (критичные / высокие / обычные) |
| 9 | План / факт по ролям | Сравнение план/факт часов в разрезе ролей за квартал |
| 10 | Часы по типам (8 недель) | Стек-диаграмма недельных трудозатрат по категориям |
| 11 | Топ рисков | Список ключевых рисков с уровнем, эффектом, ответственным и предлагаемым действием |
| 12 | Загрузка по ролям | Полоски «насколько загружен ресурс» по каждой роли |

### 3.1. Кнопка «Построить» / «Пересчитать»

Дашборд не строится автоматически. На первый заход за квартал нужно нажать «Построить» — система соберёт цифры, сгенерирует ИИ-сводку и сохранит результат. Дальше при заходе на страницу показывается сохранённая версия. Когда нужны свежие данные — «Пересчитать».

Сборка идёт фоновой задачей. Цифры берутся из базы быстро, ИИ-сводка — заметно дольше (запрос к языковой модели). Кнопка показывает индикатор «строится», по завершении появляется зелёное уведомление «Дашборд обновлён» и дата у шапки.

### 3.2. Четыре KPI верхнего уровня

| Карточка | Что считаем | Когда зелёная | Когда жёлтая | Когда красная |
|----------|-------------|---------------|--------------|---------------|
| **Индекс здоровья** | Сводный балл из критичности, возраста задач, плана и нагрузки | ≥80 | 60–79 | <60 |
| **Загрузка ресурса** | Средняя загрузка по ролям за квартал, в процентах | 70–100% | <70% | >100% (перегруз) |
| **Критичные риски** | Открытые задачи с приоритетом Critical / Highest / Blocker | 0 | 1–3 | ≥4 |
| **Выполнение сценария** | Факт-часы делённые на план утверждённого сценария, % | ≥90% | 70–89% | <70% |

Загрузка >100% всегда красная — это перегрузка ресурса, бьёт в риски.

### 3.3. ИИ-сводка из трёх секций

Под KPI — три цветные плашки:
- **«Что улучшилось»** (зелёная) — что стало лучше за квартал, краткими тезисами.
- **«Где риск»** (жёлтая) — на что обратить внимание.
- **«Что делать»** (серая) — конкретные действия, которые рекомендуются.

Сводка генерируется языковой моделью на основе цифр со всей страницы. Если ИИ выключен в настройках — наверху появится синяя плашка «ИИ выключен администратором», текст сводки останется от последней удачной генерации (помечен как «fallback» если сборка падала и подставлен запасной текст).

### 3.4. Тренд здоровья

График-«заливка» за последние 8 недель. Показывает, как менялся индекс здоровья. Полезно увидеть, что цифра 78 — это либо устойчиво так, либо мы сейчас в провале по сравнению с прошлой неделей.

### 3.5. Здоровье модулей

Список функциональных модулей (например, «Учёт», «Зарплата», «Розница»). По каждому:
- цветная точка — зелёный / жёлтый / красный;
- название и короткая заметка;
- степень риска текстом (например «нет блокеров», «3 инцидента»);
- полоска загрузки в процентах.

Модули и пороги настраиваются на бэкенде. Это не «модули Jira», а бизнес-области, по которым агрегируется отчёт.

### 3.6. Очередь по типам

Горизонтальная стек-диаграмма. По каждому типу задач (например, инциденты / доработки / консультации) — сколько штук с какой критичностью открыто. Видно, что 80% «красного» — это инциденты, и куда уходит фокус команды.

### 3.7. План / факт по ролям

Парные столбики «План» (бирюза светлая) и «Факт» (бирюза тёмная) по каждой роли (Аналитик / Разработка / Тестирование / ОПЭ). Это та же цифра, что лежит в Capacity, но агрегированная по кварталу одним графиком.

### 3.8. Часы по типам — 8 недель

Стек-диаграмма недельных трудозатрат по категориям задач: инциденты, доработки, консультации, регламент. Видно, куда уходят часы команды и как это менялось.

### 3.9. Топ рисков

Список плашек с цветной полоской слева:
- название риска;
- эффект (как он бьёт);
- ответственный + предлагаемое действие;
- тег уровня — «Высокий» / «Средний» / «Низкий».

Топ-N выбирается автоматически из открытых задач и просроченных квартальных целей. Если рисков не выявлено — плашка «Рисков не выявлено».

### 3.10. Загрузка по ролям

Для каждой роли — полоска текущей загрузки в процентах. Цвет:
- бирюза — норма (<90%);
- жёлтый — близко к пределу (90–100%);
- красный — перегруз (>100%).

Это тот же показатель, что в KPI «Загрузка ресурса», только разобранный по ролям.

## 4. Откуда цифры

- **Источник всех KPI и графиков** — те же worklog, категории и сценарии, что и в Аналитике / Capacity / Бэклоге, но агрегированные на верхний уровень. Никакой отдельной базы у дашборда нет.
- **Тренды (8 недель)** считаются ретроспективно по выбранному кварталу. Если выбран первый квартал года, в тренд попадают недели декабря-марта.
- **Загрузка по ролям** берётся из реестра ролей в Планировании и текущего активного сценария. Если активного сценария нет — план = 0 → цифры выполнения пустые.
- **Выполнение сценария** — отношение факт-часов к плановым из одного утверждённого сценария на квартал. Сценариев несколько → берётся помеченный как утверждённый.
- **Критичные риски** — открытые задачи с приоритетом Critical / Highest / Blocker, попавшие под глобальный фильтр команды.
- **ИИ-сводка** генерируется отдельной фоновой задачей. Цифры на странице есть сразу после сборки, текст ИИ-сводки может появиться через минуту-две. Дата сборки указана в шапке.
- **Глобальный фильтр команды и пикер периода** — общие для всего приложения, переключение на дашборде синхронно с другими разделами.
- **«Заменили SLA/релизы на загрузку и план-факт сценария»** — в первой версии дашборда KPI «SLA» и «Релизы» были оставлены пустыми (нет источника данных). Сейчас вместо них «Загрузка ресурса» и «Выполнение сценария» — они полностью считаются из имеющихся данных.

## 5. Типовые задачи

### Сценарий А: понедельничный обзор

1. Открыть `/executive`.
2. Проверить, что в шапке выбран нужный квартал и команда.
3. Если дата сборки старше пятницы — нажать «Пересчитать».
4. Посмотреть четыре KPI: что-то в красном/жёлтом?
5. Прочитать ИИ-сводку: «Что улучшилось» — на радостях разослать команде; «Где риск» — на повестку.
6. По графику тренда здоровья — растём, падаем, плато?
7. Топ рисков — внести 1–2 пункта в обзор недели.
8. На этом обзор за пять минут собран.

### Сценарий Б: подготовка к ПМ-комитету

1. Открыть `/executive`, переключить на отчётный квартал.
2. Нажать «Пересчитать», дождаться зелёного уведомления.
3. Сделать скриншоты:
   - KPI ряд (четыре карточки) — слайд «Состояние».
   - План/факт по ролям — слайд «Загрузка».
   - Топ рисков — слайд «Что мешает».
4. Скопировать текст ИИ-сводки «Что делать» — это готовая выжимка решений.
5. Если нужны детали по конкретной цифре — перейти в Аналитику с тем же пикером периода в шапке.

### Сценарий В: разобраться, почему «загрузка ресурса» красная

1. KPI «Загрузка ресурса» = 115% (красный).
2. Прокрутить вниз до блока «Загрузка по ролям» — увидеть, какая роль уходит за 100%.
3. Перейти в `/capacity`, оставить тот же квартал и команду — посмотреть, кто конкретно перегружен.
4. При необходимости — в `/resource-planning` сценарий перераспределить нагрузку.

### Сценарий Г: разобраться с критичными рисками

1. KPI «Критичные риски» = 7 (красный).
2. Спуститься к блоку «Топ рисков» — прочитать список с эффектом и ответственным.
3. Кликнуть по интересному пункту (если открывается deep-link в Jira) — провалиться в задачу.
4. По итогам встречи назначить действия ответственным, занести их в Jira; на следующей сборке цифра должна снизиться.

## 6. Ограничения

- **Дашборд не строится автоматически.** Каждый заход — это «прочитать сохранённую сборку». Свежие цифры — только после нажатия «Пересчитать».
- **ИИ-сводка асинхронна.** Цифры обновляются быстро, текст ИИ — отдельной задачей. Если ИИ выключен или упал — будет fallback-текст или старый.
- **Один утверждённый сценарий на квартал.** KPI «Выполнение сценария» считается по одному сценарию. Если их несколько утверждённых — берётся последний; если ни одного — KPI пустой.
- **Тренды считаются за 8 недель.** Это окно не настраивается из интерфейса. Длиннее периоды — смотрите в Аналитике.
- **Здоровье модулей и топ рисков** — на бэкенде есть конфигурируемые пороги. Что считать «жёлтым», что «красным» — заложено в коде, через интерфейс пока не меняется.
- **Глобальный фильтр команды** определяет, чьи цифры собираются. Если фильтр пуст — собираются все команды.
- **Дашборд кэшируется** на 60 секунд. Если только что нажали «Пересчитать» и кэш не успел обновиться — может показать предыдущее значение, перезагрузить страницу.

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| ИИ-сводка пустая или старая, наверху плашка «ИИ выключен администратором» | В Settings → ИИ выключен общий тумблер | Попросить администратора включить ИИ, после этого «Пересчитать» |
| KPI «Выполнение сценария» = 0% или пусто | Нет утверждённого сценария на выбранный квартал в Планировании | Зайти в `/planning`, утвердить сценарий или создать новый |
| Цифры на дашборде не сходятся с Аналитикой | Разный период или фильтр команды | Сверить глобальный пикер периода и фильтр команды в шапке — они должны совпадать с теми, что стоят на странице Аналитики |
| KPI «Загрузка ресурса» красная (>100%) | План в утверждённом сценарии больше доступной ёмкости команды | Перейти в `/capacity` и `/resource-planning`, перераспределить нагрузку или пересмотреть сценарий |
| Тренд здоровья прыгает | За 8 недель попали разные кварталы или большая синхронизация изменила историю | Проверить, не было ли полной перезагрузки ворклогов в последний период |
| «Топ рисков» пустой, хотя в Jira есть критичные задачи | Глобальный фильтр команды отсёк эти задачи, либо у них не проставлена категория | Снять фильтр команды и пересчитать; если помогло — добавить нужную команду в фильтр |
| Кнопка «Построить» сама не запускается | Это нормально — дашборд не строится по таймеру | Нажать «Построить» вручную; для автоматизации есть отдельная фоновая задача в `/sync` |
| Нажал «Пересчитать», висит долго | ИИ-провайдер отвечает медленно, либо в Settings → ИИ выбрана медленная модель | Подождать; если падает с ошибкой — проверить настройки провайдера в Settings → ИИ |
| Дата сборки старая, кнопка не нажимается | ИИ выключен — мутация заблокирована | Включить ИИ в настройках |
| «Очередь по типам» показывает 0 по всем типам | Нет задач, попадающих под фильтр команды и квартал | Проверить, что задачи синхронизированы и категоризованы в `/categories` |

## 8. Связанные разделы

**Откуда приходят данные:**
- **Синхронизация** (`/sync`) — без свежих ворклогов и задач дашборд показывает старое.
- **Категоризация** (`/categories`) — без категорий часть задач не попадает в KPI и в топ рисков.
- **Capacity** (`/capacity`) — источник плана и факта по ролям.
- **Планирование** (`/planning`) — источник сценария для KPI «Выполнение сценария».
- **Settings → ИИ** — включает/выключает ИИ-сводку и выбирает провайдера/модель.
- **Settings → Категории** — определяет категории, по которым строится «Часы по типам».
- **Глобальный фильтр команды** и **глобальный пикер периода** в шапке — общие для дашборда и всех остальных страниц.

**Куда уходит результат:**
- Никуда дальше дашборд не отдаёт данные — это конечная точка для руководителя.
- ИИ-сводку можно скопировать руками в отчёт / письмо / ПМ-комитет.

Дашборд — это вершина пирамиды: всё, что показано тут, рождается в других разделах. Если на дашборде цифра «не та», лечить нужно в источнике, не в дашборде.
