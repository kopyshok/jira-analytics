# Синхронизация с Jira

Раздел `/sync` — единая точка, через которую система забирает свежие данные из Jira: задачи, ворклоги, статусы, иерархию. Без свежей синхронизации все остальные разделы — Дашборд, Аналитика, Планирование, Capacity — показывают вчерашние или более старые цифры.

---

## 1. Зачем раздел

Аналитика, план/факт и сценарии строятся на копии данных Jira, лежащей в нашей локальной базе. Сама Jira в режиме реального времени никем не опрашивается — это медленно и упирается в её ограничения. Поэтому раз в N часов мы вытаскиваем изменения и сохраняем у себя.

Если синхронизация не запускается:
- В Дашборде «зависает» вчерашняя картина — новые ворклоги не учитываются, закрытые задачи всё ещё в работе.
- В очереди «К разбору» (`/categories`) не появляются новые задачи — аналитик не видит, что прилетело.
- В Capacity план/факт сотрудника отстаёт ровно на «возраст» последней синхронизации.
- В сценариях (`/resource-planning`) даты и оценки расходятся с реальностью.

Синхронизация работает в трёх режимах разной глубины (быстрый / обычный / полный) и в двух форматах вызова (вручную или по расписанию). Этот раздел — единственное место, где запускают, останавливают, расписывают и отлаживают синхронизацию.

---

## 2. Когда сюда заходить

- Утром перед началом разбора задач — убедиться, что ночной авто-синк прошёл успешно (вкладка «Синхронизация» → лента запусков).
- Когда коллега залогировал ворклог только что и хочется увидеть его в Аналитике немедленно — запустить быстрый режим.
- После того как в Jira прошли массовые изменения статусов, переезды задач между эпиками или массовое добавление команд — запустить обычный режим.
- Когда в Jira удалили старый ворклог, а у нас он всё ещё «висит» в фактах — это лечится только полной перезагрузкой с вкладки «Дополнительно».
- При подозрении на «потерянный» день ворклогов — ручной backfill с конкретной даты (вкладка «Дополнительно»).
- При настройке нового авто-расписания или включении/выключении существующего — вкладка «Расписание».
- После большой выгрузки в Аналитике, чтобы убедиться, что данные на момент отчёта актуальны.

## 3. Элементы интерфейса

| № | Элемент | Что показывает |
|---|---------|----------------|
| 1 | Вкладка «Синхронизация» | Ручной запуск + лента всех запусков (ручных и авто) |
| 2 | Вкладка «Расписание» | Список авто-запусков по расписанию (cron-выражения), управление ими |
| 3 | Вкладка «Дополнительно» | Ручной backfill ворклогов с произвольной даты + полная перезагрузка ворклогов |
| 4 | Карточка «Запуск синхронизации» (вкл. 1) | Кнопка «Запустить» + выпадающий список режимов + (при необходимости) выбор команды |
| 5 | Индикатор этапов (вкл. 1) | Список стадий запуска со статусом (выполняется / готово / ошибка) и счётчиками |
| 6 | Кнопка «Прервать» (вкл. 1) | Останавливает текущий запуск; уже сохранённые данные остаются |
| 7 | «История запусков» (вкл. 1) | Таблица последних 30 запусков с режимом, статусом и источником (ручной / авто) |
| 8 | Раскрытие строки в истории | Показывает этапы запуска, их длительность и текст ошибки, если есть |
| 9 | «Расписание автозапуска» (вкл. 2) | Таблица расписаний: название, выражение времени, режим, переключатель «Включено» |
| 10 | «Добавить» расписание (вкл. 2) | Открывает форму нового расписания |
| 11 | «Запустить» на строке расписания (вкл. 2) | Немедленный запуск по правилам этого расписания, не дожидаясь следующего срабатывания |
| 12 | «Обновить ворклоги с даты» (вкл. 3) | Догружает ворклоги, начиная с выбранной даты, не трогая остальное |
| 13 | «Полная перезагрузка (удалить и перечитать)» (вкл. 3) | Удаляет все ворклоги с выбранной даты и перечитывает их из Jira с нуля |
| 14 | Чекбокс «Включить выбранные команды» (вкл. 3) | Ограничивает backfill ворклогов командами, выбранными в Категоризации |

Подробности по каждому пункту — ниже в подразделах 3.1–3.6.

### 3.1. Режимы запуска

В выпадающем списке кнопки «Запустить» доступно четыре режима. Они отличаются тем, что именно тянется из Jira и сколько это занимает:

| Режим | Что делает | Когда выбирать |
|-------|------------|----------------|
| **Быстрый (ворклоги)** | Догружает только новые/изменённые ворклоги за последние дни | Самый частый сценарий: «коллега залогировал, хочу видеть в аналитике»; используется в авто-расписании по умолчанию |
| **Обычный (задачи + ворклоги)** | Обновляет задачи (статусы, поля, родительские связи, иерархия) и подтягивает ворклоги | После массовых изменений в Jira; перед выпуском еженедельного отчёта |
| **Полный (всё заново)** | Перечитывает ВСЕ задачи и ВСЕ ворклоги за установленный период с нуля | Раз в квартал или при подозрении, что локальная база рассинхронизировалась; занимает 10+ минут |
| **По команде** | То же что обычный, но только по задачам выбранной команды | Когда нужно срочно подтянуть данные одной команды, не дожидаясь полной обработки всех |

Режим «По команде» требует выбрать команду из выпадающего списка справа. Если команд нет — режим недоступен (в Jira не настроено поле команды или в системе не подтянуты значения).

### 3.2. Индикатор этапов и прерывание

Любой запуск состоит из нескольких этапов (задачи, ворклоги, иерархия и т. п.). Каждый этап показывается своей строкой:
- **крутящийся индикатор** — этап выполняется сейчас;
- **зелёная галочка** — этап успешно завершён, рядом — счётчики (сколько задач, сколько ворклогов);
- **красный крестик** — этап упал, рядом — короткий текст ошибки.

Полоса прогресса вверху — общая, она показывает только «идёт работа», без процента (точный процент по Jira не считается). Кнопка «Прервать» останавливает запуск на ближайшей безопасной точке. Уже сохранённые данные остаются — потерянных частичных обновлений не будет, в истории такая строка получит статус «Прерван».

### 3.3. Лента запусков

Таблица «История запусков» показывает последние 30 запусков, отсортированных по времени. Колонки:
- **Время** — когда запустился;
- **Режим** — быстрый / обычный / полный / по команде;
- **Команда** — для режима «По команде»; для остальных — прочерк;
- **Статус** — успешно / частично / ошибка / прерван / выполняется;
- **Источник** — «Авто» (по расписанию) или «Ручной» (с этой страницы);
- **Завершён** — момент окончания.

Клик по строке раскрывает детали: список этапов, длительность каждого, текст ошибки. Если в системе есть запуск в статусе «Выполняется» — лента сама обновляется каждые 5 секунд. Кнопка «Обновить» сверху — ручное обновление списка.

### 3.4. Расписание автозапусков

Расписание — это правило вида «каждый день в 06:00 запусти быстрый режим». Таблица показывает все правила, переключатель «Включено» гасит/возобновляет конкретное правило, не удаляя его. По умолчанию настроено одно правило — быстрый авто-синк каждые 2 часа.

Кнопка «Запустить» на строке расписания выполняет это правило прямо сейчас (полезно для проверки, что правило настроено корректно, не дожидаясь следующего времени срабатывания). Корзина с подтверждением — удаление правила насовсем.

Форма «Добавить» спрашивает:
- **Название** — для удобства, чтобы понимать «что это за правило»;
- **Cron-выражение** — стандартный формат времени, например `0 6 * * *` (каждый день в 06:00) или `0 */2 * * *` (каждые 2 часа). Подсказка под полем помогает;
- **Режим** — какой режим запускать (быстрый / обычный / полный / по команде);
- **Команда** — обязательно для режима «По команде», пусто для остальных;
- **Включено** — стартовать ли расписание сразу после создания.

### 3.5. Backfill ворклогов с даты

Вкладка «Дополнительно» нужна для ручных операций с ворклогами, когда обычная синхронизация не покрывает кейс. В шапке — выбор даты (по умолчанию подставляется ранее использованная дата).

**«Обновить ворклоги с даты»** — догружает ворклоги, у которых дата (started) с выбранного дня и позже. Уже существующие в нашей базе ворклоги при этом обновляются (если в Jira их поправили), удалённые в Jira — **остаются у нас**. Если рядом стоит чекбокс «Включить выбранные команды» — обновление ограничивается этими командами (берётся из выбора команд в Категоризации).

Используется, когда:
- кто-то залогировал ворклог задним числом, и обычная синхронизация его не подхватила;
- после перерыва (отпуск, праздники) нужно одной операцией догнать целый период.

### 3.6. Полная перезагрузка ворклогов

**«Полная перезагрузка (удалить и перечитать)»** — это единственный способ убрать из нашей базы ворклоги, которые удалили на стороне Jira. Кнопка красная и под подтверждением намеренно: она сначала удаляет все ворклоги с выбранной даты, потом перечитывает их из Jira заново.

Используется, когда:
- в Jira удалили или массово переписали старые ворклоги, и наш Дашборд показывает «ушедшие» цифры;
- расходится план/факт в Capacity у конкретного сотрудника, и есть ощущение, что у нас лишние данные.

В повседневке этой кнопкой пользоваться не нужно — она перетряхивает большой объём данных.

## 4. Откуда цифры

- **Какие задачи тянутся** — из проектов, перечисленных в Settings → Scope. Всё, что вне scope, в систему не попадает, даже если в Jira команда есть.
- **Поле «команда»** — настраивается в Settings → Поля Jira. Туда вписан идентификатор кастомного поля Jira, в котором проставляется команда (основная и/или участвующая).
- **Иерархия задач** (родитель → ребёнок) — настраивается в Settings → Иерархия задач. От неё зависит, какие типы считаются «контейнерами», а какие — «листьями».
- **Где хранится копия Jira** — в локальной базе сервиса. После синхронизации все разделы читают именно её, а не Jira напрямую.
- **Период ворклогов** — у быстрого режима это последние несколько дней (с запасом, чтобы не упустить задним числом). У полного — весь установленный окно глубины.
- **Как часто обновляется автоматически** — по правилам расписания (по умолчанию каждые 2 часа в быстром режиме). Расписание можно переопределить.
- **Что записывается в историю** — каждый запуск, ручной он или автоматический. По истории видно, когда последний раз обновлялись данные и был ли это «быстрый» или «полный» прогон.

## 5. Типовые задачи

### Сценарий А: первичная настройка после развёртывания

1. Открыть Settings → Подключение, ввести креды Jira, проверить связь.
2. Settings → Scope: добавить проекты, которые система должна видеть.
3. Settings → Поля Jira: указать поле команды.
4. Settings → Иерархия задач: убедиться, что правила соответствуют принятым в команде типам.
5. Открыть `/sync`, вкладка «Синхронизация», выбрать режим **«Полный»** и нажать «Запустить». Первичная загрузка займёт от 10 минут до часа в зависимости от размера scope.
6. Дождаться статуса «Успешно» в ленте запусков.
7. Перейти в `/categories` — разобрать стек новых задач.
8. Открыть вкладку «Расписание», убедиться что включено правило быстрого авто-синка (или создать своё).

### Сценарий Б: восстановление после потери ворклогов

Признаки: в Дашборде «провал» по часам за конкретный день; в Capacity у сотрудника план/факт расходится с реальностью.

1. Открыть `/sync` → вкладка «Дополнительно».
2. Выбрать дату на 1-2 дня раньше «провала».
3. Нажать **«Обновить ворклоги с даты»**. Если backfill не помог — значит, в Jira ворклог был удалён или сильно изменён.
4. Тогда — **«Полная перезагрузка (удалить и перечитать)»** с той же даты. Подтвердить.
5. Дождаться окончания (полоса прогресса + счётчики этапов).
6. Перепроверить Дашборд и Capacity.

### Сценарий В: ручной backfill за период

Кто-то вернулся из отпуска и залогировал ворклоги задним числом за две недели.

1. `/sync` → вкладка «Дополнительно».
2. Выбрать дату — первый день отпуска.
3. (По желанию) поставить галочку «Включить выбранные команды», если хочется ограничить операцию своей командой.
4. Нажать **«Обновить ворклоги с даты»**.
5. Дождаться окончания — в уведомлении видны цифры по сегментам A и B (ворклоги в scope и за пределами).

### Сценарий Г: проверка авто-синка

1. `/sync` → вкладка «Синхронизация», лента запусков.
2. Отфильтровать взглядом строки с источником «Авто», посмотреть их статус. Если последний авто-запуск старше суток или в статусе «Ошибка» — что-то не так с расписанием.
3. Открыть вкладку «Расписание», проверить переключатель «Включено» нужного правила.
4. Нажать «Запустить» на строке расписания — проверить, что правило исполняется.
5. Вернуться в ленту запусков — там появилась новая строка от «Авто», статус «Выполняется», следом «Успешно».

## 6. Ограничения

- **Прервать = безопасно, но не атомарно**. Уже записанные за время запуска данные остаются. Если прервали полную перезагрузку на середине — часть ворклогов уже стёрта и перечитана, часть — нет. Лучше дождаться окончания или запустить заново.
- **«Обновить ворклоги с даты» не удаляет** ворклоги, исчезнувшие в Jira. Чтобы убрать «осиротевшие» ворклоги — только полная перезагрузка.
- **Быстрый режим не подтягивает изменения задач** (статусы, поля). Если в Jira массово переехали задачи в новый эпик — нужен обычный или полный.
- **Один запуск одновременно**. Если хотите запустить вручную, пока идёт авто-запуск, кнопка «Запустить» будет ждать его завершения.
- **Cron-выражение интерпретируется по серверному времени**. Если сервер живёт в UTC, а команда — в Москве, время нужно пересчитывать (например, `0 3 * * *` = 06:00 МСК для UTC-сервера).
- **Удаление расписания** — без отмены. Удалили — придётся создавать заново.
- **Поля Jira, которых нет в Settings → Поля Jira**, не синхронизируются. Если в команде завели новое кастомное поле и хотите его видеть в аналитике — сначала указать его идентификатор в настройках, потом синхронизироваться.
- **Scope — это фильтр на входе**. Задачи проектов вне scope не появятся даже после полной перезагрузки. Чтобы добавить проект — Settings → Scope → синхронизация в режиме «Полный».
- **Категории не пересчитываются автоматически после синхронизации**. Если правила маппинга в Settings → Категории работ менялись — нужно вручную нажать там «Пересчитать маппинг по задачам».

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| После «Запустить» сразу красный крестик на первом этапе | Креды Jira просрочились или поменялись | Settings → Подключение, обновить токен, проверить связь, повторить запуск |
| В Дашборде «зависла» вчерашняя картина, цифры не двигаются | Авто-расписание выключено или последний запуск с ошибкой | `/sync` → вкладка «Расписание»: проверить переключатель. Вкладка «Синхронизация»: посмотреть статус последнего «Авто» в ленте |
| Запустил «Быстрый», но новых задач в `/categories` нет | Быстрый режим не тянет задачи — только ворклоги | Перезапустить в режиме «Обычный» |
| В Capacity у сотрудника план/факт показывает лишние часы | В Jira ворклог удалили/переписали, у нас он остался | Вкладка «Дополнительно» → «Полная перезагрузка» с даты, начиная за день до проблемной |
| Запуск висит в «Выполняется» больше часа | Скорее всего, упёрся в лимиты Jira либо сервер недоступен | «Прервать», подождать 5 минут, запустить заново. Если повторяется — проверить статус Jira |
| После добавления проекта в Scope он не появился | После изменений в Scope нужна синхронизация — Scope только описывает, а не тянет | `/sync` → режим «Полный» (для первой загрузки нового проекта), потом обычный по расписанию |
| Расписание создано, но не срабатывает | Переключатель «Включено» выключен, либо ошибка в cron-выражении | Проверить переключатель. Cron-выражение из 5 полей: минута, час, день, месяц, день недели |
| Backfill ворклогов с даты «не видит» новые ворклоги | Дата выбрана позже, чем дата нового ворклога; либо ворклог в команде, не входящей в фильтр | Сдвинуть дату раньше; снять галочку «Включить выбранные команды» |
| Прервал «Полная перезагрузка» на середине — теперь часть ворклогов потеряна | Уже стёртые данные не возвращаются, надо перечитать заново | Запустить полную перезагрузку повторно с той же даты, дождаться окончания |
| После синхронизации в Аналитике странные категории | Маппинг категорий не пересчитан после изменений в правилах | Settings → Категории работ → «Пересчитать маппинг по задачам» |

## 8. Связанные разделы

**Что должно быть настроено ДО синхронизации:**
- **Settings → Подключение** — креды Jira. Без них синхронизация падает на первом этапе.
- **Settings → Scope** — какие проекты Jira видит система. Что не в scope — не появится никогда.
- **Settings → Поля Jira** — какие кастомные поля Jira (команда, цели, оценки) подтягиваются.
- **Settings → Иерархия задач** — какие типы задач считаются «родителями». Меняет структуру дерева в Категоризации и Аналитике.

**Что использует результат синхронизации:**
- **Категоризация задач** (`/categories`) — стек «К разбору» наполняется новыми задачами после каждой синхронизации.
- **Дашборд** (`/`) — KPI и виджеты считают часы из загруженных ворклогов.
- **Аналитика** (`/analytics`) — иерархический отчёт работает на свежей копии Jira.
- **Проекты** (`/projects`) — карточка проекта, AI-саммари — всё на копии данных.
- **Capacity** (`/capacity`) — план/факт по сотрудникам читается из ворклогов.
- **Бэклог инициатив** (`/backlog`) — задачи с категорией «Инициативы (на потом)» подтягиваются по результатам синхронизации.
- **Планирование** (`/planning`, `/resource-planning`) — оценки и иерархия инициатив берутся из локальной копии Jira.
- **Executive Dashboard** (`/executive`) — все KPI верхнего уровня посчитаны на снимке, оставленном последней синхронизацией.

Старая, упавшая или забытая синхронизация → старые или неверные цифры во всех перечисленных разделах. Если что-то «не сходится» — первое место, куда нужно посмотреть, это лента запусков на `/sync`.
