# Настройки сервиса

Раздел `/settings` — административная панель. Здесь настраивается подключение к Jira, какие проекты учитывать, какие поля считать «командой» и «целями», как строить иерархию задач, справочники категорий, причин отсутствий, видов работ, производственный календарь, ИИ-функции, видимость пунктов меню и пользователи.

---

## 1. Зачем раздел

Сервис собирает данные из Jira и работает поверх собственных справочников. Без них аналитика не запустится: дерево задач не построится, ворклоги не лягут в категории, ресурсы не будут считаться. «Настройки» — это место, где администратор один раз настраивает все правила и периодически возвращается, чтобы добавить нового пользователя, обновить календарь на следующий год или подкрутить промпт для ИИ.

Все вкладки внутри раздела самостоятельные, открываются по очереди. Активная вкладка зашита в адресе страницы — ссылку на нужный экран можно скопировать и переслать коллеге.

---

## 2. Когда сюда заходить

- Первый запуск сервиса — заполнить «Подключение к Jira», выбрать проекты, сопоставить ключевые поля.
- После обновления Jira — если завели новое поле для команды/целей/часов, нужно зайти в «Поля Jira» и привязать его.
- Раз в год (декабрь–январь) — обновить производственный календарь на следующий год.
- Когда из аналитики выпадает или, наоборот, попадает что-то лишнее — проверить scope проектов и правила иерархии.
- Когда сотрудник присоединился к компании — завести его в «Пользователи».
- Когда ИИ-саммари в разделе «Проекты» работают плохо или их вообще не должно быть — вкладка «AI».
- Когда руководство просит «спрячьте этот раздел из меню» — вкладка «Видимость».

---

## 3. Элементы интерфейса

Сверху — шапка раздела «Настройки», под ней — горизонтальные вкладки. Кнопка «Справка» — этот документ. Каждая вкладка живёт сама по себе, перейти между ними можно в любой момент, несохранённые правки на одной вкладке не уносят правки на другой.

| № | Вкладка | Что настраивается |
|---|---------|-------------------|
| 1 | Подключение к Jira | Адрес, учётная запись и токен для доступа к Jira Cloud |
| 2 | Проекты в scope | Какие проекты Jira попадают в аналитику |
| 3 | Поля Jira | Какие пользовательские поля Jira считать «командой», «целью», «оценкой» и т. д. |
| 4 | Правила иерархии | Какие типы задач считать контейнерами в дереве (эпик, инициатива, …) |
| 5 | Причины отсутствий | Справочник причин для разделов «Ресурсы» / Capacity |
| 6 | Категории работ | Справочник категорий + кнопка пересчёта маппинга по задачам |
| 7 | Виды работ | Справочник видов работ (анализ, разработка, тестирование, …) |
| 8 | Производственный календарь | Праздники, переносы, нормы часов; синхронизация с РФ-календарём |
| 9 | AI | Включение/выключение ИИ, выбор провайдера, ключи, промпт, запасные модели |
| 10 | Видимость разделов | Какие пункты бокового меню скрыть |
| 11 | Пользователи | Учётные записи сотрудников, роли, команды (только для администратора) |

Подробности по каждой вкладке — ниже.

### 3.1. Подключение к Jira

Три поля: адрес вашей Jira (Base URL вида `https://your-domain.atlassian.net`), e-mail пользователя-интегратора и его API-токен. Токен получается в личном кабинете Atlassian (Account Settings → Security → API tokens).

Кнопка «Сохранить» записывает значения в базу сервиса. Токен после сохранения отображается как «токен сохранён» — увидеть его обратно нельзя, можно только перезаписать новым значением.

Кнопка «Проверить подключение» делает тестовый запрос к Jira и показывает имя и e-mail подключившегося пользователя. Это рекомендуется сделать сразу после сохранения, до запуска первой синхронизации.

Учётка нужна одна на сервис — все пользователи будут видеть только те задачи, которые видна эта учётная запись в Jira. Поэтому она должна иметь доступ ко всем нужным проектам.

### 3.2. Проекты в scope

Сверху — текущий scope: цветные плашки с ключами проектов. Крестик у плашки — убрать проект из scope. Если scope пуст, синхронизация подтянет все доступные проекты целиком — это длительно и обычно не нужно.

Ниже — таблица всех проектов Jira с фильтром по команде (используется поле «команда», настроенное на вкладке «Поля Jira»). Чекбоксы — добавить/исключить проект. Изменения накапливаются в черновике; кнопка «Сохранить» применяет их пакетом.

Проекты вне scope не синхронизируются вовсе — их задачи в дерево не попадают, в аналитику не идут. Это главный фильтр сервиса.

### 3.3. Поля Jira

Список пользовательских (кастомных) полей Jira, привязанных к ролям в сервисе. Поля сгруппированы в раскрывающиеся секции:

- **Команды и цели** — поля «продуктовая команда», «участвующие команды», «цели». От первых двух зависит фильтр команды по всему сервису.
- **Описание задачи (для ИИ-саммари)** — поля «цель» и «текущее поведение» из карточки задачи. Попадают в промпт ИИ при генерации саммари проектов.
- **Плановые трудозатраты (часы)** — четыре поля «оценка часов» по ролям: анализ, разработка, тестирование, ОПЭ.
- **Вовлечённость и длительности** — задел под календарное планирование, нужны для отображения «сколько дней займёт» в разных видах работ.
- **Приоритизация** — поля Impact и Risk.
- **Оценка заказчика** — три поля 1–5 (качество, скорость, результат).
- **Плановые даты** — задел под календарное планирование.

В каждом поле — выпадающий список реально существующих в Jira пользовательских полей с поиском. Кнопка «Сохранить» сверху сохраняет все группы пакетом.

Если какие-то поля не нужны команде — можно оставить пустыми; функциональность, которая на них завязана, просто не отрендерит соответствующие колонки/виджеты.

### 3.4. Правила иерархии задач

Таблица правил, в каком порядке сервис решает, считать ли задачу «контейнером» (эпик, инициатива, фича) или обычной задачей-исполнителем. От этого зависит, как строится дерево в разделе «Разбор задач», и куда дети наследуют категорию.

Колонки:
- **Приоритет** — целое число. Меньше — раньше. Первое совпавшее правило выигрывает.
- **Проект** — конкретный проект или «любой».
- **Тип задачи** — конкретный тип Jira или «любой».
- **Без родителя** — правило срабатывает только для корневых задач (у которых нет parent link).
- **Контейнер** — «да» — считается контейнером, «нет» — обычной задачей.
- **Активно** — выключатель, не удаляя правило.
- **Описание** — комментарий для команды.

Стрелки вверх/вниз меняют приоритет правила. Кнопка «Правило» открывает форму добавления. Удалить можно красной кнопкой в строке.

Типичное использование: добавить правило «в проекте X считать Story контейнером» — все стори в этом проекте начнут разворачиваться как контейнеры.

### 3.5. Причины отсутствий

Справочник для раздела «Ресурсы» (`/capacity`). Каждая запись — это плашка причины («Отпуск», «Больничный», «Декрет», …). Колонки:
- **Code** — короткий идентификатор причины. Используется внутри системы.
- **Название** — отображаемое.
- **Плановое** — да/нет (отпуск планируется заранее, больничный — нет).
- **Цвет** — в шестнадцатеричном виде (`#rrggbb`). Влияет на цвет плашки в тепловой карте.
- **Активна** — если выключена, причину нельзя выбрать в новых записях, но старые записи остаются.

Стрелки вверх/вниз меняют порядок отображения в селекторах. Кнопка «Добавить причину» — форма создания. Удаление возможно только если причина нигде не используется.

### 3.6. Категории работ

Справочник категорий, в которые аналитик распределяет задачи в разделе «Разбор задач» (`/categories`). От категории зависит, попадёт задача в аналитику или в архив, и в какую очередь.

Таблица:
- **Перетаскивание** (значок-«держалка» слева) — менять порядок строк.
- **Code** — внутренний идентификатор. Системные категории (Архив, Инициативы, Незаполненные worklog) — нельзя удалить.
- **Название** — редактируется на месте.
- **Цвет** — палитра, открывается по клику. Цвет используется в графиках и плашках.
- **Вид работ** — привязка категории к виду работ (см. вкладку «Виды работ»). Нужна для отчёта по видам работ.
- **Системная** — плашка «да» означает, что категория базовая, её нельзя удалить.
- **Удалить** — мусорка для несистемных категорий, через подтверждение.

Внизу — кнопка **«Пересчитать маппинг по задачам»**. Запускает фоновый процесс, который пробегает по всем задачам и переприменяет правила сопоставления категорий (если что-то изменилось в правилах автомаппинга). После окончания приходит уведомление с числом обновлённых записей. Не путать с пересчётом аналитики — это именно перерасчёт самой привязки задача → категория.

Кнопка «Добавить категорию» открывает форму с названием, кодом (можно оставить пустым, сгенерируется автоматически) и цветом.

### 3.7. Виды работ

Справочник видов работ — это шире, чем категории. Категория всегда относится к одному виду работ, но видов работ может быть больше: некоторые из них применяются как «обязательные» в сценариях ресурсного планирования.

Таблица:
- **Код** — внутренний идентификатор. У системных видов кода нельзя менять.
- **Название** — отображаемое.
- **Активен** — да/нет. Неактивные не предлагаются в селекторах.
- **Вычитается из пула** — если включено, плановые часы этого вида уменьшают доступный «проектный пул» сотрудника в сценариях. Используется для административных видов работ.
- **Порядок** — сортировка в списках.

Кнопки «Добавить вид работ», «Изменить», «Удалить» — стандартные.

### 3.8. Производственный календарь

Календарь рабочих и нерабочих дней. Используется при расчёте базовых ресурсов сотрудников (план = норма по календарю минус отсутствия).

Сверху — поле с годом и две кнопки:
- **«Загрузить с xmlcalendar.ru»** — синхронизация календаря РФ с публичным источником на выбранный год. Сервис скачивает официальный календарь (праздники, переносы, предпраздничные дни) и применяет его к базе. Ручные правки администратора **не затираются** — они остаются как есть, в сводке после загрузки видно, сколько добавлено / обновлено / пропущено как ручных.
- **«Добавить день»** — ручная запись для конкретного дня (особый праздник компании, нерабочий день по приказу, и т. п.).

Таблица показывает все дни года: дата, тип (рабочий / выходной / праздник / предпраздничный / перенесённый), переключатель «Рабочий», норма часов, примечание, источник (xmlcalendar или ручной). Ручные записи можно удалить через кнопку-мусорку.

Норму часов можно оставить пустой — она пересчитается по правилу (Пн–Пт = 8 часов, предпраздничный = 7, выходной = 0). Если в день нужно проставить нестандартную норму — указать вручную.

### 3.9. ИИ-настройки

Управление ИИ-функциями сервиса: автоматические саммари проектов в разделе «Проекты», ИИ-сводка в разделе руководителя, разметка тематического отчёта.

**Кнопка выключения ИИ** в правом верхнем углу. Главный переключатель, доступный администратору. Когда ИИ выключен:
- все ИИ-вызовы заблокированы на стороне сервиса (даже если у пользователя есть кэш кнопки «Сгенерировать заново»);
- ранее сгенерированные саммари остаются видимыми, новые не появляются;
- в разделе показано предупреждение «ИИ выключен администратором».

Это нужно, например, чтобы временно отключить ИИ из-за исчерпания квоты бесплатного провайдера или по решению безопасности.

**Провайдер ИИ.** Выбор основного поставщика модели:
- **Google Gemini** — основной free-tier (рекомендуется Gemini 3.1 Flash Lite Preview).
- **OpenRouter** — десятки бесплатных моделей через общий шлюз.
- **DeepSeek** — прямой API, платный (V3.2 chat — недорогой, R1 reasoner — дороже).

Для каждого провайдера — отдельный блок с выбором модели, ключом доступа и кнопкой «Обновить список моделей» (подтягивает актуальные модели с серверов провайдера). Ключи хранятся защищённо, после ввода отображаются как точки и обратно не показываются.

**Запасные модели (fallback chain).** Только у OpenRouter. Список моделей, которые сервис перебирает по очереди, если основная модель вернула ошибку (исчерпание лимита, временная недоступность, пустой ответ). Рекомендуется выбирать модели разных провайдеров, чтобы не «упасть» все разом.

**Системный промпт саммари.** Текст, который сервис подмешивает в каждый запрос к ИИ как роль/инструкции стиля. Можно отредактировать прямо здесь. Описание формата ответа (какие поля JSON ждать от ИИ) — фиксированное, его меняет только разработчик: иначе сломается парсинг. При смене текста все ранее сгенерированные саммари помечаются устаревшими и будут перегенерированы автоматически в ближайший проход (или сразу — кнопка «Перегенерировать все саммари»).

**«Проверить подключение»** делает тестовый ИИ-запрос и показывает имя модели и провайдера, либо текст ошибки.

### 3.10. Видимость разделов

Список разделов бокового меню сервиса (Дашборд, Проекты, Аналитика, Сценарии, …), сгруппированный по блокам «ОБЗОР / ПЛАНИРОВАНИЕ / ДАННЫЕ». Чекбокс рядом с разделом — «скрыть». Кнопка «Сохранить» — применить.

Настройка глобальная: применяется ко всем пользователям. После сохранения нужно перезагрузить страницу в браузере (своей и пользователей), чтобы меню перестроилось.

Используется, чтобы временно убрать незаконченные разделы или те, которыми команда не пользуется. Содержимое разделов не удаляется, ссылки по прямому адресу продолжают работать — только пункт меню исчезает.

### 3.11. Пользователи (только администратор)

Учётные записи сотрудников, которые могут заходить в сервис.

Таблица: имя, e-mail, роль, команда по умолчанию, активен.

Роли:
- **Администратор** — полный доступ, включая раздел «Настройки».
- **Руководитель (все команды)** — видит данные всех команд, без админских правок.
- **Руководитель** — видит только свою команду (поле «Команда по умолчанию»).

Кнопки в строке:
- **Карандаш** — редактировать имя, роль, команду.
- **Ключ** — сбросить пароль (задать новый вручную).
- **Переключатель «Активен»** — выключить учётку без удаления.

Кнопка «Добавить пользователя» — форма создания: имя, e-mail, пароль (минимум 8 символов), роль, команда по умолчанию.

Команда по умолчанию задаёт стартовый фильтр команды в шапке приложения. Пользователь может переключить фильтр на любую другую — но по умолчанию увидит свою.

---

## 4. Откуда цифры

В этом разделе цифр немного — это панель настроек. Но что зависит от настроек:

- **Список доступных проектов** в scope-таблице берётся напрямую из Jira (учётка из вкладки «Подключение»).
- **Список пользовательских полей Jira** в выпадающих списках вкладки «Поля Jira» — тоже из Jira.
- **Список команд** для фильтра scope — из значений того поля Jira, которое настроено как «команда».
- **Производственный календарь по умолчанию** — с публичного API xmlcalendar.ru (РФ).
- **Список моделей ИИ** — с серверов провайдеров (Google AI, OpenRouter, DeepSeek), кнопка «Обновить список».
- **Текущий пользователь** определяет, видна ли админская вкладка «Пользователи» (см. ниже «Ограничения»).

## 5. Типовые задачи

### Сценарий А: первая настройка сервиса с нуля

1. Открыть «Подключение к Jira». Ввести адрес, e-mail, токен. «Сохранить». «Проверить подключение» — должно показать имя пользователя.
2. Открыть «Поля Jira». Развернуть «Команды и цели». Выбрать поле для продуктовой команды (поиск по названию). «Сохранить» (кнопка сверху).
3. Открыть «Проекты в scope». Выбрать команду в фильтре — отобразятся её проекты. Отметить чекбоксами нужные. «Сохранить».
4. Открыть «Производственный календарь». Выбрать текущий год. «Загрузить с xmlcalendar.ru».
5. Открыть «Пользователи». Завести учётные записи коллег.
6. Перейти в «Синхронизация» и запустить первый полный синк.

### Сценарий Б: подключить новое поле «Цели» из Jira

1. Открыть «Поля Jira».
2. Развернуть «Команды и цели».
3. В поле «Поле целей» выбрать нужное (поиск по названию).
4. «Сохранить» (кнопка сверху).
5. Запустить инкрементальный синк — поле подтянется в задачи, появится в колонке «Цели» в разделе «Разбор задач».

### Сценарий В: обновить календарь на следующий год

1. Открыть «Производственный календарь».
2. В поле «Год» ввести следующий год.
3. «Загрузить с xmlcalendar.ru». Подтвердить.
4. Уведомление покажет, сколько дней добавлено/обновлено.
5. Если в компании есть особые нерабочие дни (например, корпоративные) — добавить их вручную через «Добавить день».

### Сценарий Г: временно выключить ИИ

1. Открыть «AI».
2. В правом верхнем углу — переключатель «ИИ включён». Перевести в «выключено».
3. Все ИИ-функции в сервисе сразу станут недоступны. Старые саммари — остаются видимыми.
4. Когда лимит обновится (или вопрос решится) — перевести обратно во «включено».

### Сценарий Д: добавить нового сотрудника

1. Открыть «Пользователи».
2. «Добавить пользователя». Заполнить имя, e-mail, временный пароль, роль.
3. В «Команда по умолчанию» выбрать команду сотрудника.
4. «Создать». Сотрудник может заходить.
5. Передать ему e-mail и временный пароль. После первого входа он не сможет сам поменять пароль — это делается через «Ключ» в админке.

### Сценарий Е: пересчитать категории задач после правок справочника

1. Открыть «Категории работ».
2. Внести нужные правки (переименовать, изменить цвет, привязать к виду работ).
3. Нажать «Пересчитать маппинг по задачам».
4. Дождаться уведомления «Маппинг пересчитан».
5. Зайти в раздел «Разбор задач», проверить, что задачи распределились правильно.

## 6. Ограничения

- **Раздел доступен только администратору.** Пользователи с ролями «Руководитель» и «Руководитель (все команды)» этот раздел в меню не видят и попасть в него по прямой ссылке не могут.
- Вкладки **«Пользователи», «Обратная связь», «Использование» видны только администратору**, даже если ту же страницу `/settings` каким-то образом открыл другой пользователь.
- **Токен Jira не показывается обратно** после сохранения — только перезаписать новым.
- **Ключи ИИ-провайдеров не показываются обратно** — глазик переключает CSS-маскировку только в момент ввода нового значения.
- **Изменения видимости разделов применяются после перезагрузки страницы** — открытая в браузере вкладка продолжит показывать старое меню, пока её не обновить.
- **Удаление категории/причины/вида работ заблокировано**, если они уже используются в задачах/записях/сценариях. Сначала перепривязать данные, потом удалять.
- **Изменения текста промпта ИИ** инвалидируют все ранее сгенерированные саммари — они пересоберутся постепенно при просмотре проектов или сразу по кнопке «Перегенерировать все саммари».
- **Учётка Jira одна на сервис.** Пользователи сервиса видят только то, что видит эта учётка — права отдельных пользователей в Jira не применяются.
- **Производственный календарь — РФ.** Импорт из xmlcalendar.ru рассчитан на российский календарь. Для других стран нужно вводить даты руками.
- **Загрузка из xmlcalendar.ru не затирает ручные дни** — они остаются как есть и в сводке показаны как «пропущено как ручных».

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| «Проверить подключение» возвращает ошибку | Неверный токен, e-mail или адрес Jira; учётка отключена; SSO-блокировка | Сгенерировать новый токен в Atlassian, перезаписать. Проверить, что Base URL — без `/jira` или иных хвостов |
| Поле в выпадающем списке «Поля Jira» отсутствует | Список не подгружен с Jira | Кликнуть в выпадающий список — он подтягивается с сервера. Если поле всё ещё нет — проверить, что у учётки Jira есть права видеть это поле |
| Scope сохранил пустоту, после синка задачи всех проектов | Пустой scope трактуется как «синхронизировать всё» | Добавить нужные проекты в scope и запустить синк заново. Лишнее можно прибрать вручную в Jira-source или подождать естественного пересчёта |
| Сделал категорию «Архив», задачи не ушли в архив сразу | Маппинг по задачам не пересчитан | Нажать «Пересчитать маппинг по задачам» в «Категории работ» |
| Календарь загрузился, но дни мои ручные не обновились | Так и должно быть — ручные дни сохраняются | Если их нужно удалить, кликнуть мусорку в строке и загрузить календарь заново |
| ИИ-саммари перестали генерироваться | Превышена квота провайдера; вместо ИИ показывается старый кэш | Открыть «AI» → проверить квоту, переключиться на другую модель / провайдера, либо выключить ИИ полностью |
| ИИ выключен — но в разделе «Проекты» видно старое саммари | Кэш ранее сгенерированных саммари остаётся видимым | Это поведение по дизайну. Чтобы убрать — придётся вручную удалить запись из проекта (либо включить ИИ и перегенерировать) |
| Пользователь видит лишний раздел в меню | Не нажата «Сохранить» в «Видимость», либо у пользователя открыта старая вкладка | Сохранить настройку, попросить пользователя обновить страницу |
| Не могу удалить категорию — «используется» | На категорию завязаны задачи | Открыть «Разбор задач», перенести задачи на другую категорию, потом удалить |
| Не вижу вкладку «Пользователи» | Открыли страницу под не-админом | Зайти под учёткой администратора. Если такой не осталось — обратиться к разработчику, чтобы создал админа через миграцию |
| Загрузка календаря говорит «не удалось» | xmlcalendar.ru недоступен, или нет интернета | Подождать, попробовать снова. Альтернатива — добавить ключевые дни вручную |

## 8. Связанные разделы

**От «Настроек» зависит почти всё:**
- **Подключение к Jira** — без этого не работает ни одна вкладка «Синхронизация».
- **Проекты в scope** — определяет, какие задачи попадают в дерево разбора, дашборд, аналитику.
- **Поля Jira** — определяют поле команды (для глобального фильтра), поле целей (тег в дереве), оценки часов (для план/факта), цели/поведение (для ИИ-саммари), оценки заказчика (для разделов «Проекты» и «Дашборд»).
- **Правила иерархии** — структура дерева в `/categories` и иерархического отчёта в `/analytics`.
- **Категории работ** — справочник для разбора задач, аналитики, тематического отчёта, ресурсного планирования.
- **Виды работ** — справочник для отчёта по видам работ и обязательных работ в сценариях.
- **Причины отсутствий** — справочник для раздела «Ресурсы» (`/capacity`).
- **Производственный календарь** — основа расчёта плановых часов сотрудников в «Ресурсах».
- **AI** — управляет ИИ-саммари в `/projects`, ИИ-сводкой в `/executive`, разметкой `/analytics/work-type-report`.
- **Видимость разделов** — управляет левым меню сервиса для всех пользователей.
- **Пользователи** — кто и под какой ролью может заходить, и какую команду видит по умолчанию.

**Куда уходит результат:**
- **Синхронизация** (`/sync`) использует подключение и scope.
- **Разбор задач** (`/categories`) использует категории, правила иерархии, поле команды.
- **Дашборд, Аналитика, Тематический отчёт, Проекты, Сценарии, Ресурсы, Планирование, Сводка для руководителя** — каждый раздел опирается на тот или иной справочник из «Настроек».

Если что-то идёт не так в любом другом разделе — стоит начать диагностику с возврата в «Настройки» и проверки соответствующей вкладки.
