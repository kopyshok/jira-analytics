# Capacity сотрудников

Раздел `/capacity` — рабочее место PM/тимлида, где видно сколько часов команда **может** отработать в квартале и сколько уже отработала. Здесь же ведутся отпуска, больничные и прочие отсутствия. Базовый ресурс отсюда уходит во все остальные планировочные разделы — сценарии, ресурсный план, executive-дашборд.

---

## 1. Зачем раздел

Без актуальной картины «кто сколько часов может» планирование квартала превращается в гадание. Бэк считает доступные часы по простой формуле: производственный календарь минус отпуска и больничные конкретного сотрудника. Получается **базовый ресурс** — сколько часов человек физически может потратить на работу в этом месяце или квартале.

Раздел `/capacity` отвечает за три вещи:

1. **Состав команды** — кто работает, в какой команде, в какой роли.
2. **Отсутствия** — кто и когда выпадает (отпуск, больничный, командировка, обучение, прочее).
3. **Сравнение плана и факта** — сколько часов было доступно по календарю vs сколько реально отлогировано в Jira.

Из этого собирается план/факт по сотрудникам, разбивка по категориям и сигнал «перегружен ли человек».

---

## 2. Когда сюда заходить

- В начале квартала — проверить, что команда укомплектована, у всех проставлена роль и команда.
- Когда сотрудник уходит в отпуск или на больничный — занести отсутствие, чтобы это попало в план.
- Когда нужно понять «почему у Иванова в плане 120 часов, а должно быть 160» — обычно ответ в отсутствиях.
- В конце месяца — посмотреть план/факт, найти перегрузы (>110%) и недогрузы (<50%).
- Перед запуском нового сценария планирования — убедиться, что базовые ресурсы свежие.
- Когда тимлид жалуется «мою команду не видно в Capacity» — проверить, что у её людей проставлена эта команда.

## 3. Элементы интерфейса

Страница делится на три вкладки и общий период вверху.

### 3.1. Шапка страницы

- **Заголовок «Ресурсы команды»** — с подписью «План · факт · отпуска · правила обязательной загрузки».
- **Чекбокс «Уточнить период»** — по умолчанию выключен и страница работает с глобальным кварталом из шапки приложения. Если включить — справа появятся поля «Год» и «Квартал», и Capacity будет смотреть на выбранный период независимо от глобального. Полезно, если нужно подсмотреть прошлый квартал, не меняя период во всех разделах.
- **Глобальный фильтр по команде** — берётся из шапки приложения (общая иконка справа). Срезает и список сотрудников в табе «Команда», и сетку отсутствий, и тепловую карту.

### 3.2. Вкладка «Команда»

Основная сетка плана и факта.

| Элемент | Что показывает |
|---------|----------------|
| Колонка «Сотрудник» | Имя + селектор роли + селектор команд (мультивыбор) |
| Колонка по каждому месяцу квартала | Группа из «План», «Факт», «%» (последние две — по переключателям) |
| Колонка «Итого» | Сумма по кварталу — план, факт, % |
| Группировка по командам | Сотрудники с одной командой собраны в свёрнутую группу, наверху строка-итог |
| «Без команды» | Группа в конце списка для тех, у кого команда не проставлена |

**Управляющая панель сверху таба:**

- **Фильтр по сотруднику** — мультивыбор. Сужает список до конкретных людей, поверх глобального фильтра по команде. Запоминается между сессиями.
- **Переключатели «Факт» и «%»** — показать/скрыть колонки с фактическими часами и процентом загрузки. По умолчанию обе скрыты, чтобы сетка была уже. Запоминаются.
- **«Добавить сотрудника»** — открывает поиск по Jira-пользователям, позволяет добавить человека в систему по имени или e-mail.
- **«Авто-определить команды»** — пробегает по всем сотрудникам, у кого команда не проставлена, и пытается её угадать по полю «Команда» из задач, на которые сотрудник логировал. Результат — уведомление «назначено N, пропущено M».
- **«Свернуть все» / «Развернуть все»** — управление группами по командам.
- **«Экспорт в Excel»** — выгружает текущую сетку плана и факта в xlsx с разбивкой по категориям задач.

**Селектор роли в строке сотрудника** — у каждого сотрудника одна роль. Список ролей берётся из вкладки «Роли» (см. ниже). Роль определяет, в каких сценариях планирования человек участвует и какие правила обязательной загрузки на него действуют. Сменить роль — выбрать в селекторе, сохраняется сразу.

**Селектор команд** — мультивыбор. Сотрудник может состоять в нескольких командах, одна из них — **основная** (отмечена золотой звёздочкой). Клик по тегу команды делает её основной. Удалить команду — крестик на теге.

**Цветовые подсветки в колонке «%»:**

- **Красный** — загрузка больше 110% (перегруз: фактических часов больше доступных).
- **Зелёный** — загрузка 100–110% (план выполнен).
- **Серый** — меньше 50% (недогруз).
- **Без подсветки** — 50–100% (нормально).

Если плана по сотруднику нет (0 часов), вместо процента ставится «—».

### 3.3. Вкладка «Отсутствия»

Сверху — тепловая карта отсутствий за квартал. По вертикали — сотрудники (отфильтрованные по глобальной команде), по горизонтали — все дни квартала. Цветной квадратик в клетке = в этот день человек отсутствует по причине X (цвет = цвет причины из справочника). Серые более тёмные клетки — будни без отсутствий, светло-серые — выходные. Наведение на клетку показывает имя, причину и диапазон.

Под картой — таблица со списком сотрудников и их отсутствиями в выбранном квартале:

- **«Сотрудник»** — имя.
- **«Отсутствия в Q… …»** — теги с диапазонами «причина: дд.мм—дд.мм», цвет тега = цвет причины. Удалить — крестик на теге.
- **«Дней»** — сколько календарных дней отсутствий в этом квартале (учитывается пересечение с границами квартала).
- Кнопка **«добавить»** в строке — открывает форму добавления одного отсутствия для этого сотрудника.

**Управление над таблицей:**

- **«Массовое добавление»** — один диапазон + одна причина → выбрать нескольких сотрудников и записать всем одинаковое отсутствие. Удобно для корпоративных обучений, новогодних каникул и т.п.
- **Переключатель «Только внеплановые»** — скрывает плановые причины (отпуск), оставляет только незапланированное (больничные, форс-мажор).

Список причин (отпуск, больничный, командировка, обучение и т.п.) ведётся отдельно в разделе **Настройки → Причины отсутствий**. Там же отмечается, какие причины считаются «плановыми», и задаётся цвет каждой.

### 3.4. Вкладка «Роли»

Справочник ролей сотрудников: аналитик, разработчик, тестировщик и т.д.

Каждая роль — это:

- **Code** — внутренний идентификатор (используется в правилах планирования).
- **Название** — что видит пользователь в селекторе.
- **Цвет** — фон тега роли в отчётах.
- **«В планировании»** — выключатель: участвует ли роль в сценариях. Например, роль «Консультант» может быть в справочнике, но не участвовать в авто-распределении часов сценария.
- **«Активна»** — выключатель: показывать роль в селекторе у сотрудника.

Порядок ролей определяет порядок в селекторе у сотрудника. Меняется стрелками «вверх/вниз». Удалить роль можно только если её нет ни у одного сотрудника — иначе сначала придётся переназначить людей.

### 3.5. Копирование правил между месяцами

Правила обязательной загрузки (например, «аналитик должен 30% времени тратить на сопровождение») задаются в сценариях планирования и хранятся помесячно. Если в предыдущем месяце правила уже настроены, их можно скопировать в текущий месяц одной кнопкой, чтобы не вводить заново. Подробности — на странице сценария планирования, Capacity показывает результат уже применённых правил.

## 4. Откуда цифры

- **План (доступные часы)** = производственный календарь региона минус отсутствия сотрудника за период. Производственный календарь живёт в **Настройки → Производственный календарь** (синхронизируется с публичным RU календарём). Каждый рабочий день по умолчанию = 8 часов; предпраздничный день = 7 часов (если так настроено в календаре).
- **Факт (отлогированные часы)** = сумма ворклогов по этому сотруднику за период из Jira. Берётся после синхронизации ворклогов (`/sync` → «Синхронизация» или «Дополнительно» → backfill).
- **Роль сотрудника** — одна, задаётся в строке сотрудника. Распределение часов между видами работ (анализ, разработка, тестирование, ОПЭ) **не задаётся** в Capacity — это уже забота сценария планирования: один и тот же человек в одном сценарии может быть аналитиком, в другом — частично разработчиком.
- **Команда** — мультивыбор. Сотрудник попадает под все свои команды при фильтрации в любом разделе. Основная команда подсвечивает строку в группировке по командам.
- **Тепловая карта отсутствий** — те же отсутствия, что и в таблице ниже, нарисованные в виде «день за день».

## 5. Типовые задачи

### Сценарий А: добавить отпуск сотруднику

1. Открыть `/capacity` → вкладка «Отсутствия».
2. Найти сотрудника в таблице (можно скрыть лишних через глобальный фильтр команды или показать «Только внеплановые»).
3. В его строке нажать **«добавить»**.
4. В форме: причина (отпуск / больничный / …) и диапазон дат.
5. Сохранить — отпуск появится тегом в строке и цветной полосой на тепловой карте.
6. На вкладке «Команда» автоматически уменьшится план этого сотрудника на дни отпуска.

### Сценарий Б: массово занести корпоративное обучение

1. Вкладка «Отсутствия» → **«Массовое добавление»**.
2. Выбрать несколько сотрудников из мультиселекта.
3. Указать причину (например, «Обучение») и диапазон дат.
4. Сохранить — запись создастся каждому из выбранных.

### Сценарий В: скопировать правила прошлого месяца

1. Правила обязательной загрузки задаются в сценарии планирования (`/planning` или `/resource-planning`), не на странице Capacity.
2. В сценарии текущего квартала — кнопка «Скопировать правила из предыдущего месяца».
3. Вернуться в `/capacity` → вкладка «Команда» — план пересчитается с учётом новых правил.

### Сценарий Г: выгрузить план в Excel

1. Вкладка «Команда».
2. Применить нужные фильтры по команде и сотрудникам.
3. Кнопка **«Экспорт в Excel»** — откроется загрузка xlsx с разбивкой по сотрудникам, месяцам и категориям задач.

### Сценарий Д: найти перегрузы

1. Вкладка «Команда».
2. Включить переключатели **«Факт»** и **«%»**.
3. В колонке «%» искать красные значения (>110%).
4. Развернуть нужную команду — увидеть, кто конкретно перегружен.
5. Кликнуть по сотруднику в любом другом разделе (например, в Аналитике) — посмотреть, на что именно ушло время.

### Сценарий Е: проверить состав команды

1. Вкладка «Команда».
2. В шапке приложения снять глобальный фильтр команд (или выбрать конкретную).
3. Кнопка **«Свернуть все»** → видны только строки с командами и количеством людей в каждой.
4. Развернуть нужную — проверить роли, команды и активность.

### Сценарий Ж: добавить нового сотрудника

1. Вкладка «Команда» → **«Добавить сотрудника»**.
2. Ввести имя или e-mail (от двух символов) — система ищет в Jira.
3. Выбрать нужного из подсказок.
4. Сотрудник появится в таблице с пустыми командой и ролью — проставить.
5. По желанию — **«Авто-определить команды»**, чтобы система угадала команду по его прошлым ворклогам.

## 6. Ограничения

- **В сетке видны только активные сотрудники.** Если человека уволили / переключили в «неактивные» — он скрыт во всех вкладках Capacity (даже если у него ещё есть ворклоги в прошлом). Признак «активный» меняется в **Настройки → Пользователи** или через карточку сотрудника.
- **Роль у сотрудника одна.** Если человек частично аналитик, частично разработчик — это решается **в сценарии планирования** через распределение часов (например, 60% анализа, 40% разработки), а не двумя ролями в Capacity. Так специально, чтобы данные оставались сравнимыми между сотрудниками.
- **Распределение часов между видами работ не задаётся в Capacity.** В этом разделе только базовый ресурс (календарь минус отсутствия). Сколько часов уйдёт на анализ, разработку, тестирование, ОПЭ — определяется сценарием планирования.
- **Базовый ресурс не учитывает «частичную занятость».** Если сотрудник работает на полставки, корректировать нужно через отсутствия (например, каждый день после обеда — «частичный отпуск») или через специально заведённую причину. Поля «% занятости» нет — упростили модель.
- **«Авто-определить команды» работает только по ворклогам.** Если у сотрудника нет ворклогов в системе (только добавили или давно не логировал) — система не сможет угадать команду, он попадёт в «пропущено».
- **Тепловая карта показывает только дни выбранного квартала.** Если отсутствие выходит за границы квартала — на карте видна только пересекающаяся часть, в таблице «Дней» — тоже только пересечение.
- **Глобальный фильтр команды** срезает данные в обеих вкладках одновременно — если на «Команде» вы переключили фильтр, на «Отсутствиях» он тоже применится.
- **Экспорт в Excel** уходит от текущего бэкенда — большие команды (>200 человек) могут грузиться несколько секунд.

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| План пустой («—» у всех месяцев) | В шапке выбрана не та команда; либо у сотрудников не проставлена команда | Проверить глобальный фильтр команды; проверить колонку «Команды» в строке сотрудника |
| Перегруз (>110%) не виден ни у кого, хотя в реальности есть | Колонка «%» скрыта — нужно включить тумблер; либо стоит фильтр по сотруднику и человек с перегрузом не попал в выборку | Включить переключатель «%», убрать фильтр по сотруднику |
| Факт сильно отличается от плана | Свежей синхронизации ворклогов не было; либо сотрудник логирует на задачи, по которым не проставлена категория | Запустить `/sync` → «Синхронизация»; проверить раздел «Категоризация», очередь «К разбору» |
| План/факт не сходится с тем, что ждали по категориям | У части ворклогов задачи без категории — они валятся в «Незаполненные worklog» и не считаются в категорийном разрезе | Разобрать очередь «К разбору» в `/categories` и пересохранить |
| После добавления отпуска план не уменьшился | Период отпуска не пересекается с выбранным кварталом; либо человек неактивный | Проверить, в каком квартале сейчас Capacity (см. чекбокс «Уточнить период»); проверить статус сотрудника |
| Авто-определение команд пропустило половину сотрудников | У этих людей нет ворклогов на задачи с заполненным полем «Команда» в Jira | Назначить команду вручную в селекторе |
| В таблице «Отсутствия» вижу пусто, но на тепловой карте — закрашенные клетки | Включён фильтр «Только внеплановые», а у людей — плановые отпуска | Снять переключатель |
| Не могу удалить роль | На этой роли есть хотя бы один сотрудник | Сначала переназначить всех на другую роль, потом удалить |
| Кнопка «Экспорт в Excel» ничего не качает | Браузер блокирует выгрузку из-за блокировщика всплывающих окон | Разрешить всплывающие окна для домена приложения |
| Тепловая карта пустая, хотя отпуска заведены | Глобальный фильтр команды отсекает всех сотрудников; либо отпуска за пределами квартала | Проверить фильтр команды и текущий квартал |

## 8. Связанные разделы

**Откуда приходят данные:**

- **Синхронизация** (`/sync`) — без свежей синхронизации ворклогов факт всегда будет нулевым или устаревшим.
- **Настройки → Производственный календарь** — определяет рабочие/выходные/предпраздничные дни и часы. База для плана.
- **Настройки → Причины отсутствий** — справочник причин с цветами и признаком «плановая».
- **Настройки → Пользователи** — список сотрудников, признак «активный», деактивация.
- **Глобальный фильтр команды** (в шапке приложения) — режет, что попадает в Capacity.

**Куда уходит результат:**

- **Сценарии планирования** (`/planning`, `/resource-planning`) — базовый ресурс Capacity = верхний предел часов, которые сценарий может распределить.
- **Executive Dashboard** (`/executive`) — секция «Загрузка команды» считает агрегаты от тех же данных, что и Capacity.
- **Аналитика** (`/analytics`) — иерархический отчёт показывает факт по сотрудникам, который должен сходиться с фактом в Capacity.
- **Дашборд** (`/`) — виджет загрузки команды использует те же базовые ресурсы.
- **Раздел «Проекты»** — план/факт по проекту в карточке проекта вычисляется из тех же ворклогов, что попадают в факт Capacity.

Если в Capacity кривые цифры — каждый из перечисленных разделов покажет кривые цифры тоже. Это базовая точка, с которой начинается планирование и отчётность.
